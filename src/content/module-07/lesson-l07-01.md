## Agent 健壮性全景：为什么"能跑"≠"可靠"

M5 的 ReAct Agent（L05-02 起）和 M6 的工具箱 Agent 都能"跑通"——在正常条件下完成任务的 demo。但生产环境不是 demo：API 会限流、网络会断、模型会幻觉、工具会超时。**"能跑"和"可靠"之间，隔着一整套 Harness 工程。**

先用监控面板感受「重试 → 熔断 → 降级」在故障注入下如何联动，再进入故障分类与可靠性指标。

::interactive{type="harnessMonitor"}

### Demo vs 生产：故障无处不在

```
Demo 环境（你演示时）：
  用户提问 → LLM 推理 → 工具调用 → 返回答案 ✓

生产环境（真实用户使用时）：
  用户提问 → LLM 推理 → 工具调用 → 429 限流！
  用户提问 → LLM 推理 → 工具调用 → 超时（外部 API 慢）
  用户提问 → LLM 推理 → 幻觉工具名 → 调用失败
  用户提问 → LLM 推理 → 推理发散 → 步数耗尽
  用户提问 → 网络 503 → 完全无响应
```

### Agent 故障模式分类

| 类别 | 故障示例 | 频率 | 影响 |
|------|----------|------|------|
| **模型故障** | API 429 限流、503 服务不可用、模型幻觉 | 高 | Agent 无法推理或推理错误 |
| **工具故障** | 外部 API 超时、返回格式异常、权限不足 | 中高 | Agent 无法获取信息 |
| **网络故障** | DNS 解析失败、连接超时、TLS 错误 | 中 | 请求发不出去 |
| **逻辑故障** | 死循环、步数耗尽、上下文超限 | 低 | Agent 卡住或产出错误 |

**关键认知**：模型故障和工具故障是最高频的——它们不是"意外"，而是**常态**。你的 Agent 必须假设它们会发生，并提前准备好应对策略。

### 可靠性指标：MTBF 和 MTTR

```
MTBF (Mean Time Between Failures) = 两次故障之间的平均时间
MTTR (Mean Time To Recovery)      = 故障后的平均恢复时间

可用性 = MTBF / (MTBF + MTTR)

示例（简化：忽略修复占用时间，按日历时间粗算）：
  Agent 每天故障 3 次，每次 2 分钟恢复
  MTBF ≈ 24h / 3 = 8 小时 = 480 分钟
  MTTR = 2 分钟
  可用性 = 480 / 482 = 99.6%
```

**Agent 的 MTTR 是关键**——你无法阻止故障发生，但可以缩短恢复时间。Harness 工程的目标就是把 MTTR 从分钟级降到秒级。

### Harness 工程：给 Agent 穿上"盔甲"

Harness 不是 Agent 的功能逻辑，而是**包裹在 Agent 外面的保护和恢复机制**：

```
┌─────────────────────────────────────────────┐
│              Harness 层（本模块）              │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐       │
│  │ 重试  │ │ 超时  │ │ 熔断  │ │ 降级  │       │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘       │
│     └────────┴────────┴────────┘             │
│              ┌─────────────┐                 │
│              │ Agent Loop  │  ← M5/M6 的 Agent │
│              └─────────────┘                 │
│              ┌─────────────┐                 │
│              │ Checkpoint  │  ← 状态持久化     │
│              └─────────────┘                 │
└─────────────────────────────────────────────┘
```

| Harness 组件 | 作用 | 对应课程 |
|-------------|------|----------|
| 重试 + 退避 | 暂时性故障自动恢复 | L07-02 |
| 超时控制 | 防止永久挂起 | L07-02 |
| Checkpointing | 失败后从断点恢复 | L07-03 |
| Circuit Breaker | 防止级联故障 | L07-04 |
| 限流 + 队列 | 峰值保护 | L07-04 |
| 降级链 | 故障时切换到备用方案 | L07-05 |

### "能跑"→"可靠"的检查清单

在把 Agent 推向生产前，逐项检查：

- [ ] API 调用有超时设置（不是无限等待）
- [ ] 暂时性错误（429/5xx）有自动重试
- [ ] 重试有退避策略（不是立即重试）
- [ ] 有最大重试次数（不是无限重试）
- [ ] 工具调用有独立超时（不跟着 LLM 超时走）
- [ ] Agent Loop 有步数上限
- [ ] 长任务有 Checkpoint（失败不从头开始）
- [ ] 外部服务故障不会导致 Agent 永久卡住
- [ ] 有降级方案（主模型不可用时切备用）
- [ ] 有日志记录（出问题能排查）

> 如果以上任何一项没做，你的 Agent 就还停留在"能跑"阶段，不适合生产环境。

### 要点总结

- "能跑"的 Agent 和"可靠"的 Agent 之间隔着一整套 Harness 工程
- 四类故障模式：模型故障（最高频）、工具故障、网络故障、逻辑故障
- 可靠性 = MTBF / (MTBF + MTTR)——Harness 的目标是缩短 MTTR
- Harness 六组件：重试退避、超时控制、Checkpoint、Circuit Breaker、限流队列、降级链
