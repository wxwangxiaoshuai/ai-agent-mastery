## Debate 与多视角辩论

L11-02 的 supervisor 是"主管指挥下属干活"。但有一类问题它解决不好——**需要深度思考、容易有偏见或盲点的决策**：这个架构方案该不该上？这个结论对不对？supervisor 自己拍板容易陷入单一视角。Debate（辩论）换了个思路：**让多个 Agent 立场对立、相互质疑，逼出更稳健的答案**。这一节讲辩论的设计，以及它最难的问题——防止"群体思维坍缩"。

### 为什么需要辩论：对抗单一视角

先理解辩论解决的认知缺陷。单个 Agent（或单个 supervisor）做决策有固有毛病：

```
单视角决策的毛病：
  · 确认偏误：倾向找支持自己观点的证据
  · 过度自信：对"看似合理"的方案不深究风险
  · 盲点：想不到的角度想不到就是想不到
  · 一锤定音：没人质疑，错了也一路错到底

辩论的解法：
  · 故意设对立立场，逼各方找对方漏洞
  · 多视角相互补盲
  · 经过质疑存活下来的结论，更稳健
```

**类比**：单 Agent 决策像"一个人拍脑袋"；debate 像"红队/蓝队对抗演练"。后者慢，但结论靠谱。这是用"多花算力"换"决策质量"——不是所有任务都值得，关键决策才用。

### 哪些任务该用 Debate

debate 有成本（多个 Agent 多轮交互），不是所有任务都该用：

| 该用 debate | 不该用 |
|-------------|--------|
| 架构/方案选型（要权衡取舍） | 简单事实查询 |
| 有争议的判断（不只有标准答案） | 流程明确的执行 |
| 高风险决策（错了代价大） | 低风险、可快速试错 |
| 需要多视角补盲（技术/商业/用户角度） | 单一领域问题 |

> 判断准则：**"这个问题值得多花 3-5 倍算力换更靠谱的答案吗？"** 值得 → debate；不值得（简单查询、流程执行）→ 单 Agent 或 supervisor 够了。debate 是重武器，别杀鸡用牛刀。

### Debate 的角色设计

辩论不是"让几个 Agent 喊出不同答案"那么简单——**角色设计决定辩论质量**。三种典型角色配置：

```
配置1：正反双方（最简）
  正方 Agent：论证"该选方案 A"
  反方 Agent：论证"不该选 A（该选 B 或 A 有问题）"
  → 二元对立，适合是非题

配置2：多立场圆桌
  技术视角 Agent：从技术可行性论
  商业视角 Agent：从成本收益论
  用户视角 Agent：从用户体验论
  → 多维度，适合复杂决策

配置3：提议者+质疑者+裁判
  提议者：提出方案
  质疑者：专门找漏洞（红队）
  裁判：综合判定
  → 有明确对抗和裁决，适合需要结论的决策
```

**关键**：角色要有**真正的立场对立**，而非都是"客观分析"。如果三个 Agent 立场都一样，辩论退化成"三个人说同样的话"——这是后面的群体思维坍缩。

### 设计辩论的轮次

辩论几轮合适？这是个 trade-off：

```
轮次太少（1轮）：
  各说各话，没真正交锋 → 没辩论效果

轮次适中（2-3轮）：
  第1轮：各方亮观点
  第2轮：针对对方观点反驳
  第3轮（可选）：回应反驳或修正立场
  → 交锋充分，成本可控，生产推荐

轮次太多（5+轮）：
  陷入细节纠缠、车轱辘话 → 成本高、收益递减
  → 容易跑偏或翻脸（L11-05 病态行为）
```

```python
DEBATE_ROUNDS = 3

def debate(question, positions):
    """多立场辩论"""
    history = [{"role": "user", "content": f"辩题：{question}"}]
    arguments = {pos: [] for pos in positions}   # 每方论点

    for round_idx in range(DEBATE_ROUNDS):
        for pos in positions:
            # 让每方基于历史论点发言
            reply = agent_debate(pos, history, round_idx)
            arguments[pos].append(reply)
            history.append({"role": "user", "content": f"[{pos}] {reply}"})

    # 裁判综合
    verdict = judge_agent(question, arguments)
    return {"arguments": arguments, "verdict": verdict}
```

**每轮的角色提示词要变**：第 1 轮"亮你的观点和论据"；第 2 轮"针对其他方的观点反驳"；第 3 轮"有无可让步或修正之处"。**让辩论递进**，而非每轮都说一遍一样的话。

### 群体思维坍缩：Debate 最大的失败

辩论最容易翻车的地方——**群体思维坍缩（groupthink collapse）**：

```
健康辩论：
  正方：A 好，因为 X
  反方：X 不成立，而且 B 更好因为 Y
  正方：Y 有道理，但 A 在 Z 上更优 → 修正为"X 场景下选 A"

坍缩的辩论（病态）：
  正方：A 好
  反方：你说得对，A 确实好
  正方：我们意见一致，选 A
  → 第 2 轮就同意了，辩论形同虚设
```

**坍缩的原因**：LLM 天生"讨好"——被Prompt 成"反方"，但模型倾向认同对方、避免冲突。几轮下来全变成附和，辩论失去意义。

**防坍缩的三个手段**：

```
1. 强制对立 prompt
   不要只说"你是反方"，要说"你的任务是找出对方观点的漏洞，
   哪怕你其实同意，也要尽力反驳——这是红队演练"
   → 明确要求"质疑"，给模型不附和的"许可"

2. 每轮换防（role rotation）
   第1轮 Agent甲当正方、Agent乙当反方
   第2轮互换
   → 防止某个 Agent 固化立场后互相熟悉而坍缩

3. 引入"魔鬼代言人"（devil's advocate）
   不管其他方说啥，专门有个 Agent 的职责是"无论多数同意什么，
   我都要找出一个反对理由"
   → 永远留一个反对声音，防一致性坍缩
```

```python
# 防坍缩的魔鬼代言人轮次
def devil_advocate_round(consensus, history):
    """无论共识是啥，专门找反对理由"""
    prompt = ("你是魔鬼代言人。当前共识是：{consensus}。"
              "你的职责是找出至少两个反对这个共识的理由，"
              "哪怕牵强也要找——确保决策经过对抗检验。").format(consensus=consensus)
    objections = llm(history, prompt)
    return objections
```

> 这是 debate 工程化最关键的一点：**默认 LLM 会坍缩到附和**，你必须主动设计"对抗压力"——强制对立 prompt、轮换、魔鬼代言人。不做这些，debate 只是浪费算力的附和会。

### 裁判：把辩论收敛成结论

辩论不是目的，结论才是。辩论完要有"裁判"把多轮论点综合成可执行的结论：

```python
def judge_agent(question, arguments):
    """裁判综合各方论点产出结论"""
    summary = "\n\n".join(f"【{pos}】\n" + "\n".join(args)
                           for pos, args in arguments.items())
    resp = client.chat.completions.create(
        model="gpt-4o",   # 裁判用更强的模型
        messages=[{"role": "system", "content":
            "你是裁判。综合各方论点，给出最终结论，"
            "明确标注：采纳了哪方哪些观点、否定了哪方哪些、理由。"},
                  {"role": "user", "content": f"辩题：{question}\n\n辩论记录：\n{summary}"}],
        temperature=0)
    return resp.choices[0].message.content
```

**裁判的设计要点**：
- **用更强的模型**：综合判断比单方论证更难，裁判该是辩论里最强的"脑"
- **要求结构化输出**：不只给结论，要说明"采纳/否定了哪些、为什么"——可追溯
- **可引入人工终审**：高风险决策，裁判结论再过人工（L10-03 HITL）才执行

### Debate 的代价与边界

清醒看代价——debate 不是银弹：

```
成本：
  · 算力：N 个 Agent × M 轮 + 裁判 = (NM+1) 次 LLM 调用
  · 时间：多轮串行，延迟数倍于单 Agent
  · 复杂度：轮次多+角色多 → 控制流复杂，易跑偏

边界：
  · 简单任务别用（杀鸡牛刀）
  · 辩论太多轮会跑偏/翻脸（L11-05）
  · 群体思维不防就坍缩（白花钱）
```

**和 supervisor 的关系**：debate 不是替代 supervisor，是**在关键决策点插入**。一个 supervisor 调度的任务里，遇到"方案选型"这种关键节点，临时切到 debate 模式，辩完结论回 supervisor 继续。**P11 就是 supervisor + debate 混合**——架构设计阶段用 debate，其余阶段 supervisor 调度。

### 要点总结

- debate 解决单视角决策毛病（确认偏误/过度自信/盲点），用"多花算力换决策质量"
- 该用：方案选型、争议判断、高风险决策、需多视角补盲；不该用：简单查询、流程执行（重武器杀鸡牛刀）
- 角色设计三种：正反双方、多立场圆桌、提议+质疑+裁判——关键是立场真正对立
- 轮次 2-3 轮为宜：太少没交锋，太多跑偏翻脸；每轮 prompt 要递进（亮观点→反驳→修正）
- 群体思维坍缩是最大失败：LLM 天生讨好附和，几轮就一致
- 防坍缩三手段：强制对立 prompt、每轮轮换防、魔鬼代言人永远留反对声
- 裁判用更强模型，要求结构化结论（采纳/否定了哪些、为什么），高风险再过人工
- 代价：NM+1 次 LLM 调用、多轮串行延迟；边界：简单任务别用、轮次别多、不防坍缩白花钱
- 与 supervisor 关系：debate 不替代 supervisor，在关键决策点插入，辩完回 supervisor——P11 混合拓扑
