## 多智能体拓扑：链式、星型、网状、层级

M10 你学会用一个或多个 Agent 编排复杂控制流。但一旦 Agent 数量变多、要协作，就出现一个新问题——**它们之间怎么连**？连接的方式（拓扑）决定了协作的效率、成本和可靠性。这一节先把多 Agent 的"连接形状"理清——这是所有后续编排（supervisor、debate、对话式）的地基。

### 单 Agent 的天花板

先说清为什么需要多个 Agent。单个 Agent 不是不行，是有上限：

```
单 Agent 瓶颈：
  · 上下文压力：一个 Agent 要同时懂需求/架构/编码/测试
    → System Prompt 臃肿，职责冲突，注意力被稀释
  · 角色冲突：又要创造又要批判 → 同一个 Agent 难以既写代码又审代码（自己审自己）
  · 失败放大：一个 Agent 卡住，整个任务停摆
  · 并行性差：单 Agent 串行执行，难利用并行

多 Agent 的解法：
  · 职责分离：每个 Agent 专精一域，Prompt 聚焦
  · 角色对立：专门的"评审者"Agent 审"实现者"Agent，客观
  · 容错：一个 Agent 挂，其他可补救
  · 并行：独立子任务可并行执行
```

> 关键认知：多 Agent 不是"更多 Agent = 更强"，而是**职责分离让每个 Agent 更专、更稳**。这是出发点，后面的所有拓扑都是为"怎么组织这些专精 Agent"服务。

### 四种典型拓扑

Agent 之间的连接形状（拓扑）主要有四种。每种有不同的通信成本、容错性、适用场景：

```
1. 链式（Chain）          2. 星型（Star/Supervisor）
   A → B → C → D            A(主管)
                              / | \
                             B  C  D (下属)
   流水线，顺序传递            中心调度，下属互不通信

3. 网状（Networked）      4. 层级（Hierarchical）
   A ─ B                       A
   │ X │                       / \
   C ─ D                      B   C
                              /\  /\
                             D E  F G
   任意两两可通信              树状，分层授权
```

逐个深入。

### 拓扑一：链式（Chain）

最简单的多 Agent——A 的输出喂给 B，B 给 C，像流水线。

```
需求分析 → 架构设计 → 编码实现 → 测试验收
  (PM)      (Architect)  (Coder)   (Tester)
```

**特点**：
- 通信成本：O(n)——每相邻两个交互一次，线性
- 容错：弱——中间任一环节卡住，后面全停
- 控制流：清晰、可预测、易调试
- 并行性：差——纯串行

**适用**：流程明确的流水线任务（软件开发的瀑布模型、内容生产流水线）。
**代表**：M10 的 CrewAI sequential 就是链式。

> 局限：链式无法表达"下游要回头找上游澄清"这种反馈。一旦需要回流，链式就要升级成更复杂的拓扑。

### 拓扑二：星型（Star / Supervisor）

一个中心 Agent（supervisor/主管）调度，下属 Agent 互不直接通信，全经主管中转。

```
        Supervisor
       /    |    \
   AgentA AgentB AgentC
```

**特点**：
- 通信成本：O(n)——每个下属只跟主管交互，比网状省
- 容错：主管是单点——主管挂全崩，下属挂主管可重试/换人
- 控制流：中心化，主管全局可见，易协调
- 职责：主管"想"（分派/汇总），下属"做"（执行）

**适用**：需要全局协调、动态分派的任务（研究 Agent 分派给搜索/抓取/总结 Agent）。
**代表**：L11-02 详讲的 LangGraph supervisor、CrewAI hierarchical。

**软肋**：主管成为瓶颈——所有信息过它一手，它的带宽/智力决定上限；下属间有直接协作需求时，绕主管中转低效。

### 拓扑三：网状（Networked）

任意两个 Agent 可直接通信——平等、去中心化。

```
   A ─── B
   │ \ / │
   │  X  │
   │ / \ │
   C ─── D
```

**特点**：
- 通信成本：**O(n²)**——两两都要交互，Agent 多了通信爆炸
- 容错：强——无单点，任一 Agent 挂，其他可绕过
- 控制流：去中心化，**难预测、难调试**
- 协调：靠协议/共识，无中央指挥

**适用**：少数 Agent 间需要灵活直接交互的场景（debate 辩论、Agent 互审）。
**风险**：Agent 数量一多，O(n²) 通信爆炸——4 个 Agent 有 6 条边，10 个有 45 条，成本失控。

> 网状是"最自由也最危险"的拓扑。L11-05 会讲它带来的病态行为（死循环、群体坍缩）。生产中网状通常**限定在小规模（3-5 个）且角色明确**的场景，不滥用。

### 拓扑四：层级（Hierarchical）

树状结构，上级管下级，下级再管更下级——像组织架构图。

```
        CEO Agent
       /        \
    TeamLead1   TeamLead2
     /   \       /   \
   A     B     C     D
```

**特点**：
- 通信成本：O(n)——每层只跟上下级交互，不跨层
- 容错：中等——某子树挂了不影响其他子树
- 控制流：分层授权，每层有局部自主权
- 扩展性：好——加新子树不影响别的子树

**适用**：规模大、可分层、各子任务相对独立（大团队分多个小组）。
**代表**：复杂组织型多 Agent、CrewAI 的 hierarchical 扩展。

**优势**：比单层 supervisor 更能 scale——主管只管几个组长，而非所有下属；比网状更可控——通信不跨层。

### 通信复杂度：O(n²) 的红线

这是这一节最重要的工程认知。不同拓扑的通信成本随 Agent 数量增长差异巨大：

```
Agent 数 n │  链式/星型/层级 O(n)  │  网状 O(n²)
──────────┼───────────────────────┼──────────────
    3     │    3 条边              │   3 条边
    5     │    5 条边              │   10 条边
   10     │   10 条边              │   45 条边   ← 已失控
   20     │   20 条边              │  190 条边  ← 通信爆炸
```

```
边数增长曲线：
边数
  │                          网状 O(n²) ╱   ← 平方级飙升
  │                       ╱
  │                   ╱
  │              ╱
  │         ╱  链式/星型/层级 O(n)  ← 线性
  │    ╱
  └───────────────────── Agent 数 n
```

**工程含义**：网状拓扑的 Agent 数量必须严格控制。3-5 个还好，超过 7 个通信成本就明显拖累——不仅慢（消息多），还乱（交互路径多，难调试），还贵（LLM 调用次数平方增长）。**这是为什么大规模多 Agent 系统几乎都用层级或 supervisor，而不是扁平网状**。

### 为任务选拓扑

按任务特征匹配拓扑：

| 任务特征 | 推荐拓扑 | 理由 |
|----------|---------|------|
| 流程明确、顺序传递 | 链式 | 控制流清晰，O(n) 成本低 |
| 需全局协调、动态分派 | 星型/Supervisor | 主管统一调度，避免下属乱撞 |
| 少数 Agent 需直接互审/辩论 | 网状（小规模） | 平等交互，但要严控数量 |
| 规模大、可分层、子任务独立 | 层级 | 可扩展，避免 O(n²) |
| 需要回流/反馈（下游找上游） | 星型或层级 | 链式无法回流 |

**决策思路**：
1. **能链式就别复杂**——顺序流水线，链式最简最可控。
2. **要协调/动态分派 → supervisor**——全局视角，下属专职执行。
3. **要平等互审/辩论 → 网状，但限 ≤5 个**——超过就 O(n²) 爆炸。
4. **规模大 → 层级**——分层后每层仍是小 supervisor/链式。

### 拓扑与前述模块的关系

把拓扑放回你已经学过的东西里，会发现它们都已有影子：

```
M5 手写 ReAct Loop     → 单 Agent 自环图（一个 Agent 自己循环）
M10 LangGraph          → 拓扑的"画图工具"（任何拓扑都能画成图）
M10 CrewAI sequential   → 链式
M10 CrewAI hierarchical → 层级（supervisor 是树根）
L11-02 LangGraph supervisor → 星型
L11-03 debate          → 网状（小规模）
```

> 关键：**拓扑是抽象，LangGraph/CrewAI/AutoGen 是实现工具**。你脑子里先有拓扑形状，再用工具落地。不要被框架的 API 带着走——先想"我的多 Agent 该连成什么形状"。

### 拓扑不是一成不变的

生产系统常**混合拓扑**——不同阶段用不同形状：

```
软件开发团队（P11）：
  需求阶段 → supervisor 调度（PM 主管需求拆解）
  设计阶段 → debate（架构师 vs 评审者辩论方案）
  实现阶段 → 链式（架构→编码→测试）
  评审阶段 → 网状（Reviewer/Testers 互审）

一个任务里，按阶段切换拓扑——这是真实多 Agent 系统的复杂度
```

**启示**：别强求一个拓扑到底。不同子任务用最合适的形状，用 LangGraph 这种能表达任意拓扑的工具把它们组合起来。P11 就会做 supervisor+debate 的混合。

### 要点总结

- 多 Agent 的出发点是职责分离让每个 Agent 更专更稳，不是"更多=更强"
- 四拓扑：链式（流水线 O(n)）、星型/Supervisor（中心调度 O(n)）、网状（平等 O(n²)）、层级（树状可扩展 O(n)）
- O(n²) 红线：网状 Agent 数量必须严控（≤5），超 7 个通信爆炸——大规模系统用层级/supervisor 而非扁平网状
- 选拓扑：顺序→链式、协调/分派→supervisor、平等互审→网状(小规模)、规模大→层级
- 拓扑是抽象，框架是实现工具——先想形状再用工具落地，别被 API 带走
- 混合拓扑是常态：不同阶段不同形状，LangGraph 能表达任意拓扑组合
