## Agent 框架全景图与选型

M5 你手写过 100 行的 Agent Loop，M6 给它加了工具，M7 加了弹性，M8 加了记忆。到这一步你该发现一个趋势：**Agent 的"胶水代码"越来越长**。再往复杂里走——多步骤、分支、并行、人工介入、状态恢复——手写循环还能撑，但代码会变成一团乱麻。框架就是来解决这个的。但框架多到眼花，选错比不选更糟。这一节先把全景理清。

### 为什么需要框架：手写循环的天花板

先客观看待手写循环——它不是"错"，是有上限。看一个典型手写 Agent 长啥样，再问"哪里开始力不从心"：

```python
# 手写 Agent Loop（M5 风格）
def run_agent(question):
    messages = [system, user(question)]
    while steps < MAX:
        resp = llm(messages, tools)
        if not resp.tool_calls:
            return resp.content
        for tc in resp.tool_calls:
            result = exec_tool(tc)
            messages.append(tool_msg(result))
        steps += 1
```

当需求长成这样，手写开始痛苦：

```
痛点1：控制流复杂化
  · 某个工具失败要重试 3 次（M7）
  · 某步结果决定走 A 分支还是 B 分支
  · 多个工具要并行，部分要串行
  → while 循环 + if 嵌套，很快变成面条代码

痛点2：状态管理
  · 中途失败要从断点恢复（M7 Checkpoint）
  · 要支持人工介入：暂停→等人审→继续
  · 要重放历史步骤调试
  → 自己实现状态机 + 持久化，重复造轮子

痛点3：多 Agent 协作
  · 几个 Agent 分工，谁先谁后
  · 结果怎么汇总
  → 手写调度，控制流和业务逻辑混在一起
```

**框架的价值**：把这些"控制流、状态、调度"的通用难题**从业务代码里抽出来**，用统一抽象表达。你只写"节点做什么"，框架管"怎么串、怎么存、怎么恢复"。

### 框架的本质：控制流抽象

所有 Agent 框架本质上都在做一件事——**给"Agent 的控制流"提供一种抽象**。差异在于抽象的"形状"不同：

```
抽象形状        代表框架        适合表达
─────────────────────────────────────────────
循环/消息流     LangChain       线性对话、简单 Chain
图/状态机       LangGraph       复杂分支、循环、持久化
角色/任务       CrewAI          多 Agent 分工协作
对话/消息       AutoGen         多 Agent 对话协作
极简循环        Smolagents      轻量、少依赖
检索/索引       LlamaIndex      RAG、数据为中心的 Agent
```

**关键认知**：没有"最好"的抽象，只有"匹配你控制流形状"的抽象。线性流程用图状态机是杀鸡用牛刀，复杂分支用线性 Chain又力不从心。选型先想清楚**你的控制流长啥形状**。

### 主流框架横评

逐个理解设计哲学，而非记 API：

| 框架 | 设计哲学 | 核心抽象 | 强项 | 弱项 |
|------|---------|---------|------|------|
| **LangChain** | 组件化 Chain | LCEL 链式表达 | 生态最大、组件最多、文档多 | 抽象层多、早期 API 频繁变、复杂控制流吃力 |
| **LangGraph** | 图状态机 | StateGraph（节点+边+状态） | 复杂控制流、循环、分支、持久化、HITL | 学习曲线陡，简单场景过重 |
| **LlamaIndex** | 数据为中心 | 索引+查询+Agent | RAG/数据管道强，文档处理细致 | Agent 控制流不如 LangGraph 灵活 |
| **CrewAI** | 角色协作 | Crew+Agent+Task | 多 Agent 分工直观、上手快 | 控制流不够细，复杂编排受限 |
| **AutoGen/AG2** | 对话消息流 | Conversable Agent | 多 Agent 对话、群体讨论自然 | 对话驱动不如图驱动可控 |
| **Smolagents** | 极简代码型 | code agent（直接写代码） | 极轻量、少依赖、代码即动作 | 能力边界靠模型，复杂流支持弱 |

**一句话记忆**：
- LangChain = 大而全的组件超市
- LangGraph = 控制流的瑞士军刀
- LlamaIndex = 数据与检索的行家
- CrewAI = 角色分工的剧组
- AutoGen = 对话沟通的圆桌
- Smolagents = 轻装上阵的极简派

### 从抽象层级理解差异

框架不是平级的，它们处在不同的抽象层。理解层级能避免"拿锤子当螺丝刀"：

```
高层（声明式，贴近业务）
  │  CrewAI：定义角色和任务，流程隐式
  │  → "PM 调研 → 架构师设计 → 程序员实现"
  │
  │  LangGraph：定义图，流程显式可控
  │  → 节点 A --条件--> 节点 B --循环--> 节点 A
  │
  │  LangChain/LCEL：定义链，线性组合
  │  → prompt | model | parser
  │
低层（命令式，贴近机器）
  │  手写 Loop（M5）
  │  → while: llm → tool → ...
```

**规律**：越往高层，表达越简洁（少写控制流），但灵活性越低（流程被框架管死）；越往低层，越灵活，但要自己管更多。**选型 = 在"简洁"和"灵活"间找你需要的层**。

> 反模式：**用高层框架硬拧低层需求**。用 CrewAI 去实现"第 3 步要看第 2 步结果决定是否回到第 1 步重做"这种精细控制流——CrewAI 的角色抽象不适合表达条件循环，硬写出来又丑又脆。这种该下到 LangGraph 层。

### 选型决策框架

按控制流形状和团队需求一步步缩小范围：

```
第1问：你的 Agent 控制流长啥样？
├─ 线性、简单（A→B→C）→ LangChain LCEL 或手写（够用即可）
├─ 有分支/循环/并行/状态恢复 → LangGraph（L10-02 详讲）
├─ 多 Agent 分工、角色明确 → CrewAI（L10-04 详讲）
├─ 多 Agent 对话、群体讨论 → AutoGen
└─ 以数据/检索为中心 → LlamaIndex

第2问：团队和工程约束？
├─ 要极轻依赖、嵌入式 → Smolagents
├─ 要最大生态和组件 → LangChain
├─ 要长期可维护、复杂控制流 → LangGraph
└─ 要快速出多 Agent demo → CrewAI

第3问：框架 vs 手写的边界（L10-06 详讲）
├─ 控制流简单、就一两个工具 → 手写更清晰
├─ 控制流复杂、要持久化/HITL → 上框架
└─ 框架让你看不懂自己的 Agent → 退回手写
```

**务实建议**：
1. **默认考虑 LangGraph**：它是最能"陪你长大"的——简单场景能写简单，复杂场景能撑住，且状态/持久化/HITL 都内建。学 LangGraph 的 ROI 最高。
2. **多 Agent 角色分工明确时用 CrewAI**：如果任务天然能拆成"PM/架构师/程序员"这种角色，CrewAI 表达更直观。
3. **RAG 重的应用用 LlamaIndex**：数据管道、索引、检索它做得最细。
4. **别一上来就 LangChain 全家桶**：它的抽象层多，简单需求被它的复杂度拖累。

### 框架的"统一姿势"：都可以用图来理解

虽然各框架抽象不同，但**底层都可以用"状态图"统一理解**——这是看穿框架的关键：

```
任何 Agent 都能画成一张状态图：
  节点 = 一步操作（调 LLM / 调工具 / 转换状态）
  边   = 控制流（下一步去哪）
  状态 = 在节点间流动的数据（messages / 中间结果）

LangGraph：直接让你画这张图（显式）
CrewAI  ：隐式的图（角色串成链/树，框架替你画）
LangChain：最简的图（线性链）
手写    ：while 循环也是一个"单节点自环图"
```

> 为什么要用图统一理解：**调试和排查问题时，无论用哪个框架，都把它的执行想象成"状态在节点间流转"**。哪里卡住、哪里状态错了、哪里该走却没走——画成图一眼看出。这是贯穿后续 LangGraph 两节课（L02/L03）的思维方式。

### 不要被"框架鄙视链"带偏

最后说几个认知误区，免得选型时被带偏：

- **"用框架 = 不够底层 = 水平差"**：错。手写 100 行 Loop 不难，难的是状态持久化、HITL、重放——这些自己实现又慢又易错。框架是工程杠杆，不是水平问题。
- **"必须用最热的框架"**：框架热度变化快，**匹配场景**比"热度"重要。AutoGen 热过、CrewAI 热过，关键是你的控制流形状。
- **"框架 API 学一遍就够了"**：错。框架的抽象决定了你能表达什么、不能表达什么。学 API 是表，学抽象边界是里（L10-06 讲怎么不被框架绑架）。

### 要点总结

- 框架解决"胶水代码膨胀"：把控制流、状态、调度从业务代码抽出来
- 框架本质都是"给 Agent 控制流一种抽象"，差异在抽象形状（循环/图/角色/对话）
- 横评：LangChain 大而全、LangGraph 控制流瑞士军刀、LlamaIndex 数据检索行家、CrewAI 角色剧组、AutoGen 对话圆桌、Smolagents 极简派
- 抽象层级：CrewAI（高，声明式）> LangGraph（中，图）> LangChain（线性）> 手写（命令式）
- 选型三问：控制流形状 → 团队约束 → 框架/手写边界
- 务实默认：LangGraph 能陪你长大（ROI 最高）；角色分工用 CrewAI；RAG 重用 LlamaIndex
- 统一姿势：所有框架底层都可画成"状态图"——调试时想象"状态在节点间流转"
- 别信鄙视链：框架是工程杠杆不是水平问题；匹配场景比热度重要；学抽象边界比学 API 重要
