## Agent 系统监控与告警体系

L15-02 的优化有没有效？生产 Agent 跑得咋样？成本涨了没？——这些靠监控回答。但 Agent 监控和传统服务很不一样：传统看 QPS/延迟，Agent 还要看**质量、行为、成本**。L13-03 讲了 trace（单请求链路），这一节讲**系统级监控与告警**——从单请求到全局运维。

### Agent 监控为什么不同于传统服务

先划清和传统服务监控的差异：

```
传统服务监控（Web API）：
  · QPS、延迟 p99、错误率、CPU/内存
  · 够用：因为逻辑确定，慢了/错了就是机器问题

Agent 监控要加三类"软"指标：
  · 质量：答得对不对（传统服务没有，输出是确定的）
  · 行为：步数、工具调用、护栏触发（Agent 特有的决策行为）
  · 成本：token 消耗、每请求成本（传统服务不看单请求成本）
```

**核心差异**：传统服务的"正确"是二值（200 还是 500），Agent 的"正确"是质量光谱。**没有质量监控，Agent 可能在"都返回 200"但答案越来越烂**——传统监控发现不了这种暗降。

### 四类监控指标体系

Agent 监控的立体指标体系：

```
1. 性能指标（和传统服务重叠）
   · QPS、p50/p99 延迟、首字延迟（流式）
   · 错误率（429/超时/工具失败）
   · CPU/内存/GPU 利用率

2. 成本指标（Agent 特有）
   · 平均 token/请求、每请求成本
   · 总 token 消耗、成本趋势
   · 缓存命中率、模型路由分布

3. 行为指标（Agent 特有）
   · 平均步数、步数分布（步数暴涨=可能发散 L05-03 / L07-01）
   · 工具调用成功率、工具选择准确率
   · 护栏触发率（输入被拦/输出被改写）

4. 质量指标（最难但最关键）
   · 用户反馈率（赞/踩）
   · LLM-as-Judge 抽样评分（L13-01）
   · 幻觉率/事实错误率
   · 任务完成率
```

> 关键认知：**Agent 监控要四类齐全**。只看性能指标（传统思路）会漏掉质量暗降——Agent 可能"都 200 且快"但答案越来越差，只有质量指标能发现。这是 Agent 运维区别于传统运维的本质。

### 质量监控：最难但必须做

质量指标最难采集——Agent 输出没有"标准答案"可断言。几种方式：

```
方式1：用户反馈（直接但稀疏）
  · 每个回答带赞/踩按钮
  · 优点：真实用户视角
  · 缺点：反馈率低（多数用户不点），有偏差（满意的少反馈）

方式2：LLM-as-Judge 线上抽样（L13-01）
  · 随机抽 N% 线上请求，LLM 裁判打分
  · 优点：覆盖广、自动
  · 缺点：成本（要调裁判模型）、裁判有偏

方式3：关键事实校验（高价值场景）
  · 对关键输出做事实校验（RAG 引用溯源验证，M4）
  · 优点：准、可定位
  · 缺点：只覆盖有明确事实的输出

方式4：下游信号（间接）
  · 客服：转人工率（Agent 答不好→转人工）
  · 推荐：点击率/采纳率
  · 优点：真实业务结果
  · 缺点：延迟（要等用户行为）
```

```python
import random

def collect_quality_signals(request, response):
    """采集质量信号"""
    signals = {}
    # 用户反馈（异步，按钮点击时回传）
    # LLM-Judge 抽样（5% 流量）
    if random.random() < 0.05:
        signals["judge_score"] = llm_judge(request, response)["score"]
    # 事实校验（含引用的输出）
    if has_citations(response):
        signals["citation_valid"] = verify_citations(response)
    return signals
```

**质量监控的现实**：单一手段都不够，**组合用**——用户反馈当基准、LLM-Judge 抽样覆盖、关键事实校验高价值场景、下游信号兜底。

### 行为监控：发现 Agent "病态"

行为指标能发现 Agent 的"暗病"——表面跑着，实则病态：

```
行为异常 → 对应问题：
  · 平均步数突涨 → 可能发散/死循环（L05-03 / L07-01）
  · 工具成功率骤降 → 某工具/API 挂了（M7 容错）
  · 工具选择准确率降 → prompt/路由出问题
  · 护栏触发率涨 → 被攻击/或护栏过敏感
  · 重复调用同一工具 → 陷入循环
  · 到 max_steps 的比例涨 → 任务越来越难或 Agent 退化
```

```python
def behavior_metrics(trace):
    """从 trace 提取行为指标（L13-03 trace 的聚合）"""
    n_tools = len(trace.tool_spans)
    return {
        "steps": trace.step_count,
        "tool_calls": n_tools,
        "tool_success_rate": (
            successful_tools(trace) / n_tools if n_tools else None),
        "tools_used": set(t.name for t in trace.tool_spans),
        "repeated_tool_calls": detect_repeats(trace),
        "hit_max_steps": trace.hit_limit,
    }
```

**行为监控的价值**：**早发现退化**。Agent 没崩（错误率正常）但行为异常（步数涨、工具失败多），说明在退化——靠行为指标在用户大量投诉前发现。

### 成本监控：防账单炸弹

Agent 成本不监控会失控——一个 prompt 改动让 token 翻倍，账单涨了才知道就晚了：

```
成本监控指标：
  · 平均 token/请求（输入+输出分计）
  · 每请求成本（按模型单价算）
  · 总消耗/日/月、成本趋势
  · 缓存命中率（L15-01 语义缓存）
  · 模型分布（路由后各模型占比，L15-02）
  · 成本/租户（多租户计费，L14-05）
```

**成本告警**：
- 每请求成本突涨（如 +50%）→ 可能 prompt 变长/路由失效/缓存崩
- 某租户成本异常高 → 可能滥用/被攻击
- 日成本超预算 → 阻断或告警

> L15-02 的三角优化，**有没有效要靠成本监控量化**——上了模型路由，看每请求成本降了没；上了缓存，看命中率多少。没有监控，优化是盲调。

### SLA 定义

把监控指标凝结成 SLA（服务等级协议）——对外/对内的承诺：

```
SLA 示例（客服 Agent）：
  · 可用性：99.9%（月停机 < 43 分钟）
  · 延迟：p99 首字 < 2s，p99 完成 < 15s
  · 质量：任务完成率 > 90%，用户满意率 > 85%
  · 错误率：5xx < 0.1%

SLA 的价值：
  · 对客户：明确承诺，建立信任
  · 对内：运维目标，违反即告警
  · 对优化：有基线衡量改进
```

**SLA 不要乱定**：
- 别承诺做不到的（99.99% 可用性，LLM API 挂了你也没辙）
- 质量类 SLA 最难——要可持续采集（LLM-Judge/反馈）
- SLA 要有误差容忍（"约 90%"而非"精确 90%"，因为质量是概率的）

### 仪表盘：把指标可视化

指标要可视化才好用。Grafana 等工具搭仪表盘：

```
Agent 运维仪表盘（推荐布局）：
  顶部：SLA 达标状态（绿/黄/红）
  第一行：性能（QPS/p99延迟/错误率）
  第二行：成本（token/请求、日成本、缓存命中率、模型分布）
  第三行：行为（平均步数、工具成功率、护栏触发率）
  第四行：质量（用户反馈率、LLM-Judge 分、任务完成率）
  侧栏：实时告警列表
```

**仪表盘设计原则**：
- **一眼定位**：SLA 状态置顶，红就出事
- **分层**：性能/成本/行为/质量分块，别混
- **趋势 > 瞬时值**：看趋势线，单点值会误导
- **可下钻**：看到异常能点进去看 trace（L13-03）

### 告警规则与通知链路

监控要配告警——指标越界主动通知，而非等人发现：

```python
# 告警规则示例
ALERT_RULES = [
    {"metric": "p99_latency", "condition": "> 5000ms", "duration": "5m",
     "severity": "warning"},
    {"metric": "error_rate", "condition": "> 1%", "duration": "1m",
     "severity": "critical"},
    {"metric": "avg_tokens_per_request", "condition": "increase > 30%",
     "duration": "1h", "severity": "warning"},
    {"metric": "avg_steps", "condition": "> 8", "duration": "10m",
     "severity": "warning"},   # 步数暴涨可能发散
    {"metric": "quality_score", "condition": "< 3.5", "duration": "30m",
     "severity": "critical"},   # 质量降
    {"metric": "daily_cost", "condition": "> budget * 0.8",
     "severity": "warning"},
]
```

**告警设计的原则**：
- **有持续时间**：单点突刺不告警（瞬时抖动），持续异常才告警
- **分级**：warning/critical，不同级别不同通知方式（邮件/电话/群）
- **可执行**：告警带上下文（哪个指标、当前值、相关 trace 链接），别告"出问题了"没下文
- **防告警疲劳**：太频繁的告警会被无视——调阈值、聚合相关告警、设静默窗口

```
通知链路：
  指标越界 → 持续 N 分钟 → 触发告警
    ├─ warning：群通知/邮件
    └─ critical：电话/短信 + 自动建工单
```

### 监控的陷阱

```
陷阱1：指标太多看不过来
  · 堆 100 个指标，没人看 → 形同虚设
  · 对策：聚焦 SLA 相关的核心指标，其余存档备查

陷阱2：只盯性能不盯质量
  · "都 200 且快"但答案烂 → 传统监控发现不了
  · 对策：质量指标必须有，哪怕难采集

陷阱3：告警风暴
  · 一个故障触发 50 条告警 → 疲劳
  · 对策：告警聚合、根因关联、静默窗口

陷阱4：监控自身没监控
  · 监控挂了没人知道 → 盲飞
  · 对策：监控的监控（meta-monitoring）

陷阱5：没有基线
  · "延迟 5s 算高吗？" → 没基线无法判断
  · 对策：先建立历史基线，再定告警阈值
```

### 要点总结

- Agent 监控区别传统：要加质量/行为/成本三类软指标——传统二值正确，Agent 是质量光谱
- 四类指标：性能(QPS/延迟/错误率)、成本(token/请求/缓存命中/模型分布)、行为(步数/工具成功/护栏触发)、质量(反馈/LLM-Judge/事实校验/完成率)
- 质量监控最难但必须：用户反馈+LLM-Judge抽样+事实校验+下游信号组合，单一不够
- 行为监控发现暗病：步数涨/工具失败/护栏触发涨/重复调用→在用户投诉前发现退化
- 成本监控防账单炸弹：每请求成本/总消耗/缓存命中/模型分布；成本告警突涨即查
- SLA：可用性/延迟/质量/错误率承诺；别乱定做不到的，质量类要可持续采集
- 仪表盘：SLA状态置顶+四类分块+趋势线+可下钻trace
- 告警：有持续时间防抖动、分级、带上下文可执行、防告警疲劳（聚合/静默）
- 陷阱：指标过多、只盯性能、告警风暴、监控自身没监控、无基线
