## 成本、延迟与质量的三角优化

L15-01 的缓存/限流是"堵"和"省"的工程。这一节谈生产环境永恒的核心矛盾——**成本、延迟、质量三角**。你不能三个都要最优：要极快可能要牺牲质量、要极准可能要贵、要极省可能慢或差。这一节量化这个三角，给优化策略。

### 三角：不可能全都要

先建立模型——这三个维度几乎总是互相矛盾：

```
          质量（准/稳）
           /\
          /  \
         /    \   要质量↑ → 要强模型 → 成本↑ + 延迟↑
        /      \
       /        \
       ──────────
  成本          延迟
  （省）        （快）

任何两点优化常牺牲第三点：
  · 又快又省 → 牺牲质量（小模型）
  · 又快又准 → 牺牲成本（强模型+缓存）
  · 又准又省 → 牺牲延迟（多轮推理/批处理）
```

**核心认知**：生产优化的本质是**明确当前场景哪个维度优先，主动牺牲另两个**。客服延迟优先（牺牲点成本用强模型）、离线分析成本优先（牺牲延迟用便宜模型）、医疗质量优先（牺牲成本和延迟都要强模型多轮）。

### 量化三角：给场景打分

不能只说"客服延迟优先"，要量化。每个需求场景定三角的目标值：

```markdown
## 场景三角量化

| 场景 | 延迟目标 | 成本上限 | 质量底线 | 优先级 |
|------|---------|---------|---------|--------|
| 实时客服 | <2s首字 | 中 | 85%准 | 延迟>质量>成本 |
| 深度研究 | 分钟级可接受 | 低 | 95%准 | 质量>成本>延迟 |
| 数据分析 | 秒级 | 低 | 80%够用 | 成本>延迟>质量 |
| 医疗辅助 | 可慢 | 高 | 99%+ | 质量>>>其他 |

→ 优化策略按优先级选，不同场景不同打法
```

**量化的价值**：把"我们要又快又准又省"的空话，变成"客服<2s 85%准 中成本"的可决策目标。没有数字，优化是无依据的玄学。

### 策略一：模型路由（降本+提速的核心）

**不同任务用不同模型**——简单问题用小模型（便宜快），复杂问题才上大模型：

```
模型路由逻辑：
  用户问题 → 路由判断 → 
    ├─ 简单（FAQ/闲聊）→ 小模型（Haiku/mini）→ 快省
    ├─ 中等（一般问答）→ 中模型（Sonnet）→ 平衡
    └─ 复杂（推理/编码）→ 大模型（Opus/4o）→ 准但贵慢
```

```python
def route_model(question: str, history: list) -> str:
    """模型路由：按复杂度选模型"""
    # 规则路由（简单）
    if is_simple_faq(question):
        return "claude-haiku-4-5"   # 简单用小模型
    if is_code_or_reasoning(question):
        return "claude-opus-4-8"    # 复杂用大模型
    return "claude-sonnet-5"        # 默认中模型

    # 进阶：LLM 路由器（用很小的模型判断复杂度）
    # complexity = tiny_llm_classify(question)
    # return MODEL_BY_COMPLEXITY[complexity]
```

**路由的价值**：
- **降本**：80% 问题是简单的，用小模型省大半成本
- **提速**：小模型快，简单问题秒回
- **保质量**：复杂问题才上大模型，质量不打折

**路由的挑战**：
- **判断准**：路由判断错了（复杂问题误判简单→小模型答不好）→ 用 LLM 路由器比规则准，但多一次调用
- **回退机制**：小模型答得不好时，自动升级大模型重答

```python
def route_with_fallback(question):
    model = route_model(question)
    answer = call(question, model)
    # 质量检查：答得不好就升级
    if quality_check(answer) < threshold and model != "claude-opus-4-8":
        answer = call(question, "claude-opus-4-8")  # 升级重答
    return answer
```

> 模型路由是 Agent 降本的第一利器。多数生产 Agent 用大模型是浪费——80% 流量本该走小模型。M1 讲过模型选型，这里是它的生产化落地。

### 策略二：缓存（L15-01 语义缓存的深化）

缓存不只语义缓存一种，分多层：

```
缓存层次（命中率从高到低，价值递减）：
  1. 精确缓存：完全相同问题 → 命中率最高
     · 适合：固定查询、重复 FAQ
  2. 语义缓存：相似问题 → 命中率中
     · 适合：自然语言变体多的场景（L15-01）
  3. Prompt Caching（M3）：相同前缀缓存 → 命中率看前缀稳定性
     · 适合：system prompt 长且固定
  4. 结果缓存：工具结果缓存 → 工具调用省
     · 适合：搜索/查询结果短期复用
```

**缓存的成本-延迟-质量影响**：
- 成本↓（命中省 LLM 调用）
- 延迟↓（命中是毫秒级）
- 质量？命中返回的是历史答案——若问题有时效性，质量可能↓

**缓存的黄金法则**：**只缓存时效不敏感、非个性化、高复用的内容**。无脑缓存会返回过时/错误答案，质量崩。

### 策略三：分级推理

复杂任务一次推理不够准——用**多级推理**：快速粗答 + 按需深答。

```
分级推理：
  L1：小模型秒级粗答 → 80% 用户满意，结束
  L2：L1 答得不好/用户追问 → 中模型精答
  L3：L2 还不够 → 大模型 + 多轮推理
  
  大部分流量停在 L1（省），少数深挖到 L3（准）
```

```python
def tiered_reasoning(question):
    # L1：小模型快速答
    ans1 = call(question, "claude-haiku-4-5")
    if quality(ans1, question) >= GOOD_ENOUGH:
        return ans1   # 80% 在这里结束
    # L2：中模型
    ans2 = call(question, "claude-sonnet-5", context=ans1)
    if quality(ans2, question) >= GOOD:
        return ans2
    # L3：大模型 + 多轮
    return deep_reasoning(question)   # 少数走这
```

**价值**：把贵的 L3 只用在真正需要的少数请求上，平均成本和延迟大降。**这是"质量优先场景也能省"的关键**——不是所有请求都要最高质量，按需分级。

### 策略四：批处理与异步

非实时场景——**批量处理**省成本：

```
批处理场景：
  · 离线数据分析：1000 个文档要分析
  · 评估流水线：跑评测集（L13-02）
  · 批量摘要/分类

批处理省成本原因：
  · 批量 API 单 token 更便宜（厂商有 batch 折扣）
  · 可用便宜时段/便宜模型
  · 无并发压力（慢慢跑）
```

**批处理 vs 实时的权衡**：
- 批处理：省成本，但延迟高（用户要等批跑完）
- 适合：不急的离线任务
- 不适合：实时交互（用户等不了批）

### 策略五：Prompt 压缩与精简

Prompt 越长成本越高（token 按输入计费）。**压缩 prompt** 降本：

```
压缩手段（M3 上下文工程的延续）：
  · 历史摘要压缩（L08-02 滑动窗口+摘要）
  · 检索结果裁剪（只留 top-k，L04-04 Reranking 后裁）
  · 工具结果裁剪（大结果只留关键字段）
  · Prompt Caching（M3，相同前缀缓存计费优惠）
  · LLMLingua 等压缩技术（M3-04）
```

**压缩的成本-质量影响**：
- 成本↓（输入 token 少）
- 延迟↓（输入短处理快）
- 质量？压缩可能丢关键信息 → 要保真压缩，非暴力截断

### 策略六：蒸馏与 Speculative（进阶）

更高级的优化，知道概念即可：

```
蒸馏（Distillation）：
  · 用大模型生成大量"教师答案"
  · 微调小模型学这些答案
  · 小模型在特定任务上接近大模型质量，但便宜快
  · 适合：固定领域、量大、要长期省成本

Speculative Decoding：
  · 小模型先快速生成草稿
  · 大模型只验证/修正草稿
  · 大模型不用从头生成，快很多
  · 适合：要大模型质量但提速
```

**这些进阶策略的边界**：蒸馏要数据量和微调工程投入，speculative 要模型架构支持。**不是所有场景都值得**——量大、长期、固定领域才上，否则路由+缓存+分级推理够用。

### 优化不要破坏可维护性

三角优化有个隐藏的第四维——**可维护性**（L14-01）：

```
过度优化的可维护性代价：
  · 路由规则复杂到没人懂 → 维护噩梦
  · 多级缓存层叠加 → 缓存一致性难管
  · 为省成本到处特判 → 代码乱
  · 微调模型 → 多了个要维护的模型版本（L14-06 三维版本）

清醒：
  · 先用简单优化（路由+缓存）拿到 80% 收益
  · 复杂优化（蒸馏/speculative）只在量真的大、简单优化不够时上
  · 每个优化要算"省的成本 vs 增的维护成本"
```

> 反模式：**为省 10% 成本引入 3 套缓存 + 微调模型 + speculative**——省的钱还不够养这套复杂系统的运维。优化要算总账（省的钱 - 增的维护成本），不是 token 单价。

### 量化的优化决策

把优化决策也做成可量化的（呼应 L14-01 Trade-off）：

```markdown
## 优化方案评估
| 方案 | 成本节省 | 延迟改善 | 质量影响 | 维护成本 | 净收益 |
|------|---------|---------|---------|---------|--------|
| 模型路由 | -40% | -30% | 中性 | 低 | 高 ✅ |
| 语义缓存 | -25% | -50% | 风险(时效) | 中 | 中 |
| 分级推理 | -30% | +20% | 中性 | 中 | 中 |
| 蒸馏 | -60% | -40% | -5% | 高 | 待量大 |

决策：先上模型路由+语义缓存（低维护高收益），分级推理按需，蒸馏等量大了再说
```

### 要点总结

- 三角：成本/延迟/质量互相矛盾，生产优化是明确场景优先级主动牺牲另两个
- 量化三角：每个场景定延迟目标/成本上限/质量底线和优先级——客服延迟优先、离线成本优先、医疗质量优先
- 模型路由（降本核心）：简单用小模型复杂用大模型，80%流量走小模型；路由要准+升级回退机制
- 缓存多层：精确/语义/Prompt Caching/结果缓存；只缓存时效不敏感+非个性化+高复用，无脑缓存返过时答案
- 分级推理：L1小模型粗答→L2中模型→L3大模型多轮，贵的L3只给少数需要——质量优先场景也能省
- 批处理省成本：离线非实时用批量API折扣+便宜模型；实时交互不适用
- Prompt压缩降本：摘要/裁剪/Caching/LLMLingua；保真压缩非暴力截断，防丢关键信息
- 蒸馏/Speculative进阶：量大长期固定领域才上，否则路由+缓存够用
- 别破坏可维护性：每个优化算"省的钱-增的维护成本"净收益；先简单优化拿80%收益，复杂优化量大再上
- 优化决策量化：方案评估表算成本/延迟/质量/维护/净收益，先上高净收益的
