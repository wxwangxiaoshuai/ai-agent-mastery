## 故障应急与容灾恢复：Incident Response

L15-03 监控帮你发现问题。但发现问题只是开始——**出了故障怎么应急、怎么恢复、怎么不让它再发生**，是运维工程的硬功夫。Agent 系统的故障有自己的特点：不只服务挂了，还有"模型抽风""工具全挂""成本爆炸"这些 Agent 特有故障。这一节讲故障应急与容灾。

### Agent 故障的特异性

先看清 Agent 故障和传统服务的不同——这决定应急方式不同：

```
传统服务故障：
  · 进程挂/机器宕 → 重启/扩容
  · 依赖（DB）挂 → 切备用
  · 二值：要么能用要么不能

Agent 故障更复杂：
  1. 基础设施故障（和传统一样）：服务挂、DB 挂、网络断
  2. 模型故障：LLM API 限流/超时/服务不可用/输出抽风（质量崩）
  3. 工具故障：依赖的外部 API 挂/变慢
  4. 质量故障：服务"正常返回"但答案越来越烂（最难发现）
  5. 成本故障：账单爆炸（某 prompt bug 让 token 暴涨）
```

**关键差异**：传统故障多是"能用/不能用"二值，Agent 还有**"能用但烂"**的质量故障——监控显示正常，用户已经在骂了。应急要覆盖这类"软故障"。

### 故障分级

不是所有故障同等重要——分级决定响应力度：

```
P0（Critical）：全站不可用 / 数据泄露 / 资金损失
  · 响应：立即，全员，电话叫醒
  · 例：Agent 服务全挂、用户数据串租户、退款 bug

P1（High）：核心功能受损 / 大量用户受影响
  · 响应：小时内，核心值班
  · 例：LLM API 全限流、知识库检索崩、质量大幅降

P2（Medium）：部分功能异常 / 少数用户
  · 响应：工作日内
  · 例：某工具偶发失败、某租户慢、缓存命中率降

P3（Low）：单点小问题
  · 响应：排期修
  · 例：个别边缘 case 答错、UI 小 bug
```

**分级的价值**：把"都叫故障"变成"按影响定响应力度"。P0 全力以赴，P3 别半夜叫醒人。**不分级的后果**：每个小问题都当紧急，真正大故障来了团队已疲劳。

### 应急响应流程

一个故障从发现到恢复的标准流程：

```
1. 发现（监控告警/用户反馈/人工巡检）
   ↓
2. 确认与分级（是不是真故障、多严重）
   ↓
3. 通告（拉群/通知相关方，避免各自为战）
   ↓
4. 止血（先恢复服务，不急着找根因）
   ↓
5. 根因定位（止血后深挖为什么）
   ↓
6. 修复（彻底解决）
   ↓
7. 验证（确认恢复，指标回到正常）
   ↓
8. 复盘（Postmortem，防再发）
```

**关键原则——先止血后定位**：

```
错误做法：故障中花半小时找根因，服务一直挂着
正确做法：先快速止血（回滚/切备用/降级），服务恢复了再慢慢定位根因

止血手段（按速度优先）：
  · 回滚（L15-05）：最近改动的回退，最快
  · 切备用：主模型挂→切备用模型（M7 降级链）
  · 降级：关掉出问题的功能/工具，其余照常
  · 限流/熔断：保护下游，先牺牲部分流量保主体
```

> 这是 M7 弹性工程的运维延续——重试/降级/熔断在生产应急时是"止血工具"。先止血保服务，再定位根因彻底修。**别在着火时研究火因**。

### Agent 特有故障的应急

针对 Agent 特有故障类型的应急手段：

**模型故障（LLM API 挂/限流）**：
```
应急：
  1. 切备用模型（Opus→Sonnet→Haiku，M7 降级链）
  2. 多供应商分流（OpenAI 挂走 Anthropic）
  3. 极端时静态兜底（"服务繁忙"，保可用性）
  
预防（M7）：
  · 多供应商、降级链、Circuit Breaker
```

**工具故障（外部 API 挂）**：
```
应急：
  1. 工具降级（主工具挂切备用，M7）
  2. 临时禁用故障工具，Agent 退化为无该工具模式
  3. 对依赖该工具的任务返回"该功能暂不可用"
  
预防：
  · 工具熔断、备用工具、HITL（L10-03 危险操作可暂关）
```

**质量故障（答案烂但服务"正常"）**：
```
应急（最难，因为不易察觉）：
  1. 监控行为/质量指标异常（步数涨/反馈降，L15-03）
  2. 快速定位：最近改了啥（prompt/模型/工具）→ 回滚
  3. 临时降级到更保守配置（如关掉刚加的工具）
  
预防：
  · 质量监控（L15-03）、灰度发布（L15-05）、评测门禁（L13-02）
```

**成本故障（账单爆炸）**：
```
应急：
  1. 成本告警触发（L15-03）
  2. 紧急限流/熔断，压住消耗
  3. 定位暴涨源（哪个 prompt/租户/路由失效）→ 修
  
预防：
  · 成本预算+硬上限、每请求成本监控、配额（L14-05）
```

### 容灾切换与降级预案

大故障（整区域/整服务挂）要**容灾切换**——切到备用系统：

```
容灾架构：
  · 多区域部署：主区域挂切备区域
  · 多可用区：单 AZ 挂不影响其他
  · 数据备份：DB 定期备份 + 异地副本

切换预案（要预演，不能出事才想）：
  · 切换条件：什么时候触发（主挂多久/多严重）
  · 切换步骤：DNS 切/流量切/数据同步状态确认
  · 切换后降级：切到备可能功能不全，提前告知
  · 切回：主恢复后怎么切回（别立即切回，先观察）
```

**容灾的等级**：

| 级别 | 描述 | 成本 | 适合 |
|------|------|------|------|
| 冷备 | 备用系统不跑，出事才启 | 低 | 非核心 |
| 暖备 | 备用系统跑着但不接流量 | 中 | 中等核心 |
| 热备 | 备用实时同步随时接管 | 高 | 核心 |
| 多活 | 多区域同时跑 | 最高 | 超核心 |

> 别盲目上多活——成本极高。按业务关键性选：客服 Agent 冷备/暖备够，支付级 Agent 才考虑热备/多活。L14-01 的"按场景权衡"在这里延续。

### 降级预案：分层保命

容灾是大切换，**降级是小步保命**——出问题不切整个系统，而是关部分功能保主体：

```
降级预案（按严重度递进）：
  L1：非核心功能降级（如关掉"创意写作"模式，保问答）
  L2：模型降级（强模型→弱模型，保速度牺牲质量）
  L3：限流降级（拒绝部分请求，保住已接受的）
  L4：只读降级（只答不写，如客服不能办业务只能问答）
  L5：静态兜底（返回"服务繁忙"，保最基本的可用性）
```

**预案要预先定义**：每个降级级别的触发条件、执行方式、恢复条件——出事时照预案执行，不临场决定。**预案文档化、定期演练**（P15 会做故障注入演练）。

### 故障复盘（Postmortem）

故障恢复后最重要的一步——**复盘**，防止再发：

```
Postmortem 模板：
  1. 摘要：什么故障、影响多大、持续多久
  2. 时间线：几点发现、几点止血、几点修复
  3. 影响分析：用户数/成本/SLA 违反
  4. 根因分析：为什么发生（5 whys，深挖不止表面）
  5. 应急评估：止血快不快、预案有没有效
  6. 改进项：防止再发的具体 action（带 owner+deadline）
  7. 经验：可复用的应急经验
```

```markdown
# Postmortem：2026-07-23 客服 Agent 质量暗降

## 摘要
7/23 14:00-16:00，客服 Agent 答案质量骤降，任务完成率 90%→72%，
约 5000 用户受影响。服务"正常"（错误率 0），靠质量监控发现。

## 时间线
14:00 质量监控告警（LLM-Judge 分降）
14:05 确认 P1，拉群
14:10 止血：回滚昨天的 prompt 改动
14:15 质量恢复
14:30 验证，恢复

## 根因（5 whys）
为什么质量降？��� prompt 改了输出格式
为什么改格式没发现？→ 评测没覆盖该 case
为什么没覆盖？→ 评测集缺该类边界 case
为什么缺？→ 评测集更新流程没要求加边界 case

## 改进项
- [ ] prompt 改动必须加边界 case 到评测集（owner: A, 1周）
- [ ] 质量监控加"输出格式"专项告警（owner: B, 2周）
- [ ] prompt 灰度从 5% 提到先 1%（owner: C, 3天）
```

**复盘的文化原则**：
- **对事不对人**：复盘找系统漏洞，不找"谁背锅"——不然下次没人敢报告故障
- **找根因不找表面**：5 whys 深挖，"prompt 改错"是表面，"评测流程没要求边界 case"是根因
- **改进项要落地**：带 owner 和 deadline，否则复盘白做
- **无指责文化**（blameless）：出故障第一责任是报告，不是隐瞒

> Google 的 SRE 书核心一条：**无指责复盘**。人都会犯错，指责让人隐瞒，隐瞒让系统更脆弱。复盘找系统性原因，改进流程，而非惩罚个人。

### 故障预防：从被动到主动

最好的应急是不出故障——预防：

```
预防手段（前面模块都讲过，这里汇总）：
  · 健壮性工程（M7）：重试/降级/熔断/checkpoint
  · 评测门禁（L13-02）：改动必过评测
  · 监控告警（L15-03）：早发现
  · 灰度发布（L15-05）：小流量验证
  · 故障注入（混沌工程）：主动注入故障验证韧性
  · 容灾演练：定期演练切换预案
  · 变更管理（L14-06）：版本化、可回滚
```

**故障注入/混沌工程**：主动给系统注入故障（杀实例、延迟依赖、限流），看它扛不扛得住——比等真实故障验证更主动。P15 会做一次完整的故障注入演练。

### 要点总结

- Agent 故障更复杂：不只"能用/不能用"，还有"能用但烂"的质量故障+成本故障+模型/工具故障
- 故障分级 P0-P3：按影响定响应力度，P0立即电话叫醒，P3排期；不分级会疲劳
- 应急流程：发现→确认分级→通告→止血→根因→修复→验证→复盘
- 先止血后定位：先回滚/切备用/降级恢复服务，再深挖根因；别着火时研究火因
- Agent 特有故障应急：模型故障切备用/多供应商、工具故障降级/禁用、质量故障定位最近改动回滚、成本故障限流压消耗
- 容灾切换：多区域/AZ/备份；等级冷备/暖备/热备/多活，按关键性选别盲目多活
- 降级预案分层保命：L1关非核心→L2模型降级→L3限流→L4只读→L5静态兜底；预案预定义+演练
- Postmortem 七部分：摘要/时间线/影响/根因(5whys)/应急评估/改进项/经验
- 复盘文化：对事不对人、找根因不找表面、改进项带owner落地、无指责(blameless)防隐瞒
- 预防：健壮性M7+评测门禁+监控+灰度+故障注入+容灾演练+变更管理
