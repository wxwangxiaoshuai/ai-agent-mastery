## Agent 产品的 UX 设计

前面五节讲"后端怎么稳"。但用户不关心你的 Agent 有多高可用——他们关心"它能不能帮我做事、我能不能信任它、它错了怎么办"。**Agent UX 是技术到体验的最后一公里**。一个技术上强的 Agent，UX 烂用户还是不用。这一节讲 Agent 产品设计的三大原则：透明度、可控性、错误恢复。

### 为什么 Agent 需要专门的 UX

先看清 Agent UX 和传统软件 UX 的根本不同：

```
传统软件 UX：
  · 确定性操作：点按钮 → 固定结果
  · 用户能预期：操作前知道会发生什么
  · 错误明确：报错信息清楚

Agent UX 挑战：
  · 非确定输出：同一输入可能不同结果
  · 用户难预期：不知道 Agent 会怎么做、答得对不对
  · 可能出错（幻觉）：且错得很"像真的"
  · 黑盒感：不知道它为啥这么做
  · 信任难题：用户凭啥信一个会幻觉的系统？
```

**核心矛盾**：Agent 能力强但不可靠+不透明，用户既想要它的能力又不敢信。**UX 的任务就是化解这个矛盾**——让用户看见、能控、错了能恢复，从而敢用。

### 原则一：透明度（Show Your Work）

最根本的原则——**让用户看见 Agent 在做什么**。黑盒的 Agent 没人敢信：

```
不透明（差）：
  用户问 → 转圈 30 秒 → "答案是 X"
  · 用户：它在干嘛？为啥 30 秒？X 哪来的？对吗？

透明（好）：
  用户问 → "正在检索相关文档..." → "找到 3 篇" → 
  "基于这些[引用1][引用2]综合" → 逐字输出答案
  · 用户：知道它在检索、来源是啥、可信
```

**透明度的层次**：

```
1. 过程可见：让用户看到 Agent 的中间步骤
   · "正在搜索" "找到3条" "正在综合"
   · L10-05 流式输出的 UX 价值

2. 来源可溯：答案标注依据
   · 引用来源（Perplexity 的核心差异化，L14-04）
   · 用户能点进去验证

3. 推理可见：复杂决策展示推理
   · "我选方案A因为成本更低（对比B）"
   · 别只给结论，给"为什么"

4. 状态可见：当前在哪一步、还要多久
   · "步骤3/5，预计还需30秒"
   · 长任务尤其重要（L15-01 异步队列的 UX）
```

```jsx
// 透明度 UX 示例（流式 + 步骤 + 引用）
function AgentResponse() {
  return (
    <div>
      {/* 过程可见：流式展示中间步骤 */}
      <Steps>
        <Step icon="🔍">检索到 3 篇相关文档</Step>
        <Step icon="📖">读取核心内容</Step>
        <Step icon="✍️">正在综合答案...</Step>
      </Steps>
      {/* 来源可溯：答案带引用 */}
      <Answer>
        根据2024年数据，X 指标增长了 15%<Cite src="doc1" />
        主要原因是 Y<Cite src="doc2" />。
      </Answer>
    </div>
  )
}
```

**透明度的边界**：不是把所有内部推理都倒给用户——那信息过载。**透明度是"用户需要的可见"**：来源要看见、状态要看见、决策依据要简述；内部 token、步数细节不必给。

> 透明度建立信任。用户不见过程就当黑盒，黑盒错了不知道为啥，下次就不敢用。**Show Your Work 是 Agent 产品的信任基石**。

### 原则二：可控性（给用户恰当下放控制权）

Agent 太自主用户会失控感；完全受控又失去 Agent 价值。**恰当下放控制权**：

```
该给用户的控制：
  · 纠正方向：Agent 走偏了用户能拉回（"不是，我要的是..."）
  · 关键决策：重大动作前让用户确认（HITL，L10-03）
  · 参数调节：温度、风格、深度等用户可调
  · 中断：能随时停（L10-05 可中断流式）
  · 重试/换方案：不满意能再来

不该给用户的控制：
  · 内部工具选择（用户不懂哪个工具合适）
  · 每步都确认（退化成人肉操作，L10-03 过度HITL）
  · 底层 prompt 细节
```

**可控性的 UX 形式**：

```
1. 确认对话：高风险动作前
   "将发送邮件给 client@x.com，主题'...'，确认发送？"
   [发送] [编辑] [取消]

2. 中断/重做：
   [停止生成] [重新生成] [换个角度]

3. 参数调节：
   详细度: [简洁 ●━━━ 详细]
   风格: [正式] [轻松]

4. 反馈循环：
   "这个答案有用吗？" [👍] [👎]
   （反馈既改进单次，也回流评测 L13-01）
```

**可控性的原则**：**给用户"方向纠正"和"关键决策权"，不给"操作细节"**。用户要的是"能影响结果"，不是"亲手操作每步"。给太多控制 = 退化成手动工具，Agent 价值没了。

### 原则三：优雅的错误恢复

Agent 一定会错——UX 要让"错了"不变成灾难，而是顺畅的恢复：

```
糟糕的错误 UX：
  Agent 答错了 → 用户：？？？ → 用户不知道错了/怎么纠 → 放弃用

优雅的错误恢复：
  Agent 答错 → 用户"不对，我要的是X" → Agent 理解纠正 → 重答
  · 用户能轻易发现错（透明度）
  · 能轻易纠正（可控性）
  · Agent 能从纠正学习（记忆/上下文）
```

**错误恢复的 UX 设计**：

```
1. 让错误可发现
   · 引用溯源让用户能验证（发现幻觉）
   · 置信度提示："我对这个答案不太确定"
   · 对比展示：给多个候选让用户选

2. 让纠正低成本
   · "重新生成"一键
   · 部分重做：只重做错的部分，不重来全部
   · 上下文记住纠正："上次你说要X风格"（M8 记忆）

3. 让失败有出路
   · Agent 做不到时明确说"我做不到X，但能做Y"
   · 转人工（L15-03 转人工率 + 客服 UX）
   · 提供替代方案而非干瘪报错

4. 错误不丢上下文
   · 答错了，纠正后基于前面的讨论继续，不重置
   · 别"错了就从头来"
```

```jsx
// 错误恢复 UX
function errorRecovery(agentAnswer) {
  if (lowConfidence(agentAnswer)) {
    return <Hint>⚠️ 这个答案我不太确定，建议核实[Cite]</Hint>
  }
}
// 用户纠正
function onUserCorrect(correctInput) {
  // 记住纠正，基于它重答，不重置上下文
  agent.regenerateBasedOn(correctInput, { keepContext: true })
}
```

> 错误恢复的精髓：**承认 Agent 会错，让"错了"成为对话的一部分而非终点**。用户说"不对"，Agent 接着纠正继续——像人对话一样。这种 UX 让 Agent 的"不可靠"变得可接受。

### 流式与延迟的 UX

L10-05 讲了流式技术。这里讲它的 UX 价值：

```
延迟的 UX 痛点：
  · >2秒无反馈：用户焦虑"卡住了？"
  · >10秒：用户离开

流式 UX 解法：
  · 首字快：用户看到开始输出 → 不焦虑
  · 步骤可见：长任务有进度 → 可预期
  · 可中断：发现跑偏能停 → 不浪费

长任务的 UX（L15-01 异步）：
  · 给 task_id + 预计时间
  · 完成通知（别让用户干等页面）
  · 进度可见（步骤3/5）
```

**延迟分级 UX**：

| 延迟 | UX 策略 |
|------|---------|
| <1s | 直接出结果，无需提示 |
| 1-3s | 转圈或"思考中" |
| 3-10s | 流式输出+步骤可见 |
| >10s | 异步+通知+进度 |

### 信任建立与预期管理

Agent 产品的长期 UX——**建立信任 + 管理预期**：

```
信任建立：
  · 一致性：同样问题答得差不多（别今天行明天不行）
  · 可解释：错了能解释为啥、能改进
  · 边界诚实：做不到的明确说做不到（别装能）

预期管理（别过度承诺）：
  · 别宣传"AI 万能" → 用户发现不能就崩
  · 明确能力边界："我擅长X，Y可能不准"
  · 示例引导：展示典型用法，让用户知道怎么用最好
```

**过度承诺的反模式**：宣传"AI 助手什么都能"→用户拿它当百科+医生+律师→各种幻觉翻车→信任崩塌。**诚实的能力边界**比"什么都能"更能建立长期信任。

### 个性化与记忆的 UX

M8 记忆系统在 UX 上的体现——**让用户感觉"它懂我"**：

```
记忆的 UX：
  · 主动确认："我记得你对花生过敏，推荐避开的花生菜，对吗？"
    （让用户知道 Agent 记了啥，也允许纠正）
  · 个性化：基于历史调整回答风格/深度
  · 可管理：用户能看/删自己的记忆（数据主权，L14-06）

记忆 UX 的坑：
  · 别"记了不该记的"（如临时情绪）→ 用户毛骨悚然
  · 记错要能纠（L08-05 记忆冲突）
  · 给用户"忘记我"的控制
```

### Agent UX 的反模式

```
反模式1：黑盒结果
  · 只给答案不给过程/来源 → 用户不敢信
  · 对策：透明度三件套（过程+来源+推理）

反模式2：过度自主
  · Agent 自己做完所有事用户没法插手 → 失控感
  · 对策：可控性（关键决策确认+中断+纠正）

反模式3：错误即终点
  · 答错就完事，用户没法纠 → 弃用
  · 对策：错误恢复（可发现+可纠+有出路+不丢上下文）

反模式4：过度承诺
  · "AI 万能" → 翻车信任崩
  · 对策：诚实边界

反模式5：记了不该记的
  · 用户随口说的被长期记 → 毛骨悚然
  · 对策：记忆可管理+主动确认

反模式6：延迟无反馈
  · 长时间转圈 → 焦虑离开
  · 对策：流式+步骤+进度
```

### M15 收官：从能部署到能运维到好体验

M15 六节串起来，生产化的完整图景：

```
L15-01 生产架构：网关/队列/缓存/限流 —— 基础设施
L15-02 三角优化：成本/延迟/质量 —— 性能经济
L15-03 监控告警：质量/行为/成本/性能 —— 可见性
L15-04 故障应急：分级/止血/容灾/复盘 —— 韧性
L15-05 灰度发布：独立灰度/A/B/回滚 —— 安全变更
L15-06 产品 UX：透明/可控/错误恢复 —— 用户体验

能力链：能部署(L01)→能省钱提速(L02)→能看见(L03)→能扛事(L04)→能安全改(L05)→能被信任使用(L06)
```

> M15 是"从能部署到能运维到好体验"的完整跨越。前 14 模块你造出了强大的 Agent；M15 让它**生产可用、经济、可观测、韧性强、安全变更、用户爱用**。P15 把这套基线落地。

### 要点总结

- Agent UX 挑战：非确定输出+可能幻觉+黑盒+信任难题；UX 化解"强能力但不可靠不透明"的矛盾
- 原则一透明度(Show Your Work)：过程可见+来源可溯+推理可见+状态可见——信任基石，但只给用户需要的，别信息过载
- 原则二可控性：给方向纠正和关键决策权(HITL/中断/重做/参数)，不给操作细节——给太多控制=退化成手动工具
- 原则三错误恢复：错误可发现(引用/置信度)、纠正低成本(重做/部分重做)、失败有出路(转人工/替代)、不丢上下文——让错了成为对话非终点
- 流式与延迟UX：<1s直出/1-3s转圈/3-10s流式步骤/>10s异步通知进度
- 信任与预期：一致性+可解释+边界诚实；别过度承诺"AI万能"——诚实边界比全能更能建长期信任
- 记忆UX：主动确认记了啥+可纠+可管理(数据主权)；别记不该记的
- 反模式：黑盒结果/过度自主/错误即终点/过度承诺/记不该记/延迟无反馈
- M15收官：能部署→能省钱提速→能看见→能扛事→能安全改→能被信任使用；P15落地基线
