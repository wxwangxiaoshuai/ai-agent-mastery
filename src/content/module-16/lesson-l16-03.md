## 模型定制化概览：微调、LoRA、DPO 的工程决策

前面 15 个模块，Agent 的能力都靠"不改模型"实现——Prompt 工程、RAG、工具、记忆。但有一种情况这些都不够：**你想让模型在某个领域/风格上深度定制**。这时候要动模型本身——微调。这一节速览微调/LoRA/DPO/RLHF 的定位，并建立"该不该微调"的工程决策框架。

### 不改模型能走多远

先认清——绝大多数场景不需要微调：

```
不改模型的手段（前 15 模块）：
  · Prompt 工程（M2）：调指令/示例改输出
  · 上下文工程（M3）：组装动态信息
  · RAG（M4）：注入外部知识
  · Few-shot：给示例教行为
  · 工具（M6）：扩展能力
  · 记忆（M8）：个性化

这些能解决：
  ✓ 知识更新（RAG）
  ✓ 行为指导（Prompt/Few-shot）
  ✓ 能力扩展（工具）
  ✓ 个性化（记忆）

这些解决不了（才考虑微调）：
  ✗ 特定领域风格深度定制（医疗/法律/某行业话术）
  ✗ 让模型稳定输出某格式/某行为（Prompt 约不住）
  ✗ 降低特定任务的成本（小模型微调后替代大模型）
  ✗ 让模型掌握 Few-shot 教不会的深层模式
```

**核心判断**：微调是重投入手段，**前 15 模块的"不改模型"手段用尽后再考虑**。多数 Agent 项目根本不需要微调——Prompt+RAG+工具覆盖了 90%+ 需求。

### 微调家族速览

不深究数学，理解各手段的定位和适用：

```
1. 全量微调（Fine-tuning）
   · 改模型所有参数
   · 效果最强但最贵（要算力+数据+工程）
   · 适合：领域极大差异、有充足数据算力

2. LoRA（低秩适配）
   · 不改原模型，加少量"适配参数"训练
   · 便宜、快、原模型可复用
   · 适合：资源有限、多任务（每个任务一个LoRA）
   · 主流微调方式

3. DPO（直接偏好优化）
   · 不靠人工打分，用"偏好对"（A比B好）训练
   · 比 RLHF 简单
   · 适合：对齐人类偏好、风格控制

4. RLHF（人类反馈强化学习）
   · 人工打分→奖励模型→强化学习
   · 效果好但最复杂最贵
   · 适合：对齐高质量、有团队和预算
```

| 手段 | 成本 | 效果 | 数据需求 | 适合 |
|------|------|------|---------|------|
| 全量微调 | 最高 | 最强 | 大 | 领域差异大 |
| LoRA | 低-中 | 中-强 | 中 | 资源有限/多任务 |
| DPO | 中 | 偏好对齐 | 偏好对 | 风格/偏好 |
| RLHF | 最高 | 好 | 人工标注 | 高质量对齐 |

**一句话记忆**：
- 全量微调 = 重型武器，效果最强代价最大
- LoRA = 主流性价比之选
- DPO = 对齐偏好的轻量手段
- RLHF = 最复杂最贵的对齐

### 微调解决什么 vs 解决不了什么

清醒划清微调的能力边界：

```
微调能解决：
  ✓ 领域风格/话术定制（医疗报告、法律文书格式）
  ✓ 稳定输出某格式（Prompt 约不住的）
  ✓ 特定任务性能提升（在该任务上比通用模型强）
  ✓ 降本：小模型微调后接近大模型效果，省成本

微调解决不了：
  ✗ 知识更新（知识会过时，RAG 才实时）
  ✗ 实时信息（微调是离线的）
  ✗ 工具能力（微调不给你调 API 的能力）
  ✗ 大幅跨领域能力（微调是窄的，专一域）
  ✗ 完全消除幻觉（微调后仍会幻觉）
```

**黄金法则（M4 说过，这里强化）**：
- **知识用 RAG**（实时、可溯源、更新便宜）
- **行为用微调**（稳定格式、领域风格）
- **小文档用长上下文**（直接塞）
- 三者不互斥，组合用

### 该不该微调的决策框架

这是本节的核心——一个清晰的决策树，避免"该不该微调"拍脑袋：

```
Step 1：Prompt/RAG/工具/Few-shot 用尽了吗？
├─ 没 → 先把这些做到位（90% 场景在这解决）
└─ 用尽了，仍不满足 ↓

Step 2：不满足的是哪类？
├─ 知识不够新/全 → RAG（不是微调）
├─ 行为不稳/格式不守 → 微调（这个微调擅长）
├─ 能力缺（调外部） → 工具/MCP（不是微调）
└─ 领域风格深度定制 → 微调 ↓

Step 3：有微调的投入吗？
├─ 有数据（足够高质量领域数据）
├─ 有算力/预算
├─ 有维护能力（模型要随业务更新重训）
│   ├─ 都有 → 微调
│   └─ 缺一 → 别微调，回到 Prompt/RAG
```

**ROI 评估表**（L14-01 的量化思路）：

| 方案 | 解决程度 | 开发成本 | 维护成本 | 风险 | ROI |
|------|---------|---------|---------|------|-----|
| Prompt 优化 | 中 | 极低 | 低 | 低 | 高 |
| RAG | 高（知识） | 中 | 中 | 低 | 高 |
| Few-shot | 中 | 低 | 低 | 低 | 中 |
| LoRA 微调 | 高（风格/格式） | 中高 | 高（重训） | 中 | 待评估 |
| 全量微调 | 最高 | 极高 | 极高 | 高 | 通常不值 |

> 决策：**ROI 排序 Prompt优化 > RAG > Few-shot > LoRA > 全量微调**。从上往下试，前面能解决就不往下走。微调几乎从不是第一选择。

### 微调的隐藏成本

别只看"能微调"，看清长期成本：

```
微调的真实成本：
  1. 数据成本
     · 要高质量领域数据（不是随便抓的）
     · 数据标注/清洗投入
  2. 算力成本
     · 训练要 GPU
     · 多次实验调参
  3. 维护成本（最易被低估）
     · 业务变了要重训（不像改 Prompt 一行就改）
     · 维护一个"定制模型"版本（L14-06 三维版本多了一维）
  4. 评估成本
     · 要建评测集验证微调有效没退化（L13-01/02）
  5. 锁定风险
     · 微调的模型绑特定基座，基座升级迁移成本高
  6. 通用能力退化风险
     · 微调窄域可能损害通用能力（catastrophic forgetting）
```

**最大陷阱——维护成本**：很多人以为微调是"一次性投入"，实则业务一变就要重训，且维护定制模型版本很重。**Prompt 改一行就改了，微调重训要数据+算力+评估全套**。

### 微调 vs RAG vs Prompt 的取舍实例

用具体场景帮判断：

```
场景1：客服要答公司产品问题
  → RAG（知识实时、可溯源）—— 别微调

场景2：要输出固定格式的医疗报告
  → 先 Prompt+Few-shot 约束，约不住再微调

场景3：让模型用某行业黑话/风格
  → Few-shot 给示例，不够再 LoRA 微调

场景4：降低高频简单任务成本
  → 模型路由用小模型（L15-02）；量大且固定再微调小模型

场景5：让模型掌握 Few-shot 教不会的深层模式
  → 微调（这是微调的强项）

场景6：让模型知道最新政策
  → RAG（微调的知识会过时）
```

**规律**：知识/实时→RAG；行为/风格→Prompt→Few-shot→微调；能力→工具。**绝大多数落在 RAG+Prompt，微调是少数派**。

### 微调的工程流程（速览）

如果真要微调，流程大致：

```
1. 数据准备（最关键）
   · 收集领域数据（问答对/偏好对/文本）
   · 清洗、去重、去噪
   · 划训练/验证集
2. 选基座+方法
   · 选开源基座（如 Llama/Qwen）
   · 选方法（多数 LoRA）
3. 训练
   · 配超参（lr/epoch/batch）
   · 监控 loss，防过拟合
4. 评估
   · 评测集对比微调前后（L13-01/02）
   · 查通用能力有无退化
5. 部署
   · 模型服务化（vLLM 等）
   · 监控线上效果（L15-03）
6. 持续维护
   · 业务变→重训→评估→灰度上线（L15-05）
```

**关键**：评估（step4）和持续维护（step6）是常被忽视但决定成败的。**没���测集别微调**——没法判断微调有没有效、有没有退化。

### 开源 vs 闭源基座

微调用什么基座：

```
闭源微调（如 OpenAI/Anthropic 的微调 API）：
  · 简单（API 调用）
  · 但受限（基座方决定能不能微调、怎么微调）
  · 数据可能要给基座方（合规考量）

开源基座微调（Llama/Qwen/DeepSeek 等）：
  · 完全可控
  · 数据不出门
  · 要自己有算力和部署能力
  · 主流选择（尤其企业）
```

**趋势**：开源模型能力追上来后，开源基座微调越来越主流——可控、合规、省长期成本。但门槛高（要算力+工程能力）。

### 要点总结

- 不改模型手段（Prompt/RAG/Few-shot/工具/记忆）覆盖 90%+ 需求，微调是少数派
- 微调家族：全量(最强最贵)/LoRA(主流性价比)/DPO(偏好对齐)/RLHF(最复杂)——多数选LoRA
- 微调能解决：领域风格/格式稳定/任务性能/小模型降本；解决不了：知识更新/实时/工具能力/跨域/消除幻觉
- 黄金法则：知识用RAG、行为用微调、小文档长上下文——组合不互斥
- 该不该微调决策树：先Prompt/RAG/工具用尽→不满足的是行为/风格→有数据算力维护→才微调
- ROI排序：Prompt优化 > RAG > Few-shot > LoRA > 全量微调——从上往下试，微调几乎从不是首选
- 隐藏成本：数据+算力+维护(最易低估，业务变要重训)+评估+锁定+通用能力退化
- 最大陷阱：微调不是一次性投入，维护成本远超Prompt——改Prompt一行vs重训全套
- 场景规律：知识/实时→RAG、行为/风格→Prompt→Few-shot→微调、能力→工具
- 微调流程：数据准备(最关键)→选基座方法→训练→评估(没评测集别微调)→部署→持续维护
- 开源基座微调主流：可控合规省长期成本，但门槛高要算力+工程
