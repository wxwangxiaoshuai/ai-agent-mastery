## Agent 架构决策框架：ADR、Trade-off 量化分析

前 13 个模块你学了 Agent 的所有技术部件——Prompt、上下文、RAG、工具、Loop、弹性、记忆、多模态、框架、多 Agent、评估、安全。这一模块升华：**怎么把这些部件组织成一个合理的系统**。架构师的核心能力不是"能实现"，而是"会决策"——在多个可行方案里，选出当前最合适的。这一节建立决策方法论。

### 从"能实现"到"会决策"

先区分两种能力，这是架构师的门槛：

```
实现者：关心"怎么做出来"
  · 选一个能跑的方案，写代码，跑通
  · 局限：可能选了能跑但难维护/贵/慢的方案

架构师：关心"为什么这么选"
  · 列出多个可行方案，分析各自代价，选当前最合适的
  · 关心长期：可维护、可扩展、成本、可靠性
  · 关键动作：把"为什么"写下来，让决策可追溯、可复盘
```

**核心转变**：从"选一个能跑的"到"权衡后选当前最优并记录理由"。能跑的方案多的是，但**为什么用 A 不用 B**——能答清这个，才叫架构决策。

> 为什么"记录"这么重要：半年后团队新人问"为啥用 LangGraph 不用 CrewAI"，没人记得，要么瞎猜要么推翻重来。ADR（架构决策记录）就是让每个决策**有据可查、可复盘、可纠错**。

### ADR：架构决策记录

ADR（Architecture Decision Record）是把架构决策**文档化**的标准实践。一个 ADR 回答五个问题：

```
ADR 标准结构：
  1. 标题与状态：决策标题 + 提议/接受/废弃
  2. 背景：为什么要做这个决策（问题/约束）
  3. 决策：选了什么方案
  4. 备选方案：考虑过哪些其他方案，为什么不选
  5. 后果：这个决策带来的好处和代价
```

```markdown
# ADR-001：使用 LangGraph 编排研究 Agent

## 状态
已接受（2026-07-23）

## 背景
研究 Agent 需要：多步分支（检索质量差改写重试）、并行搜索多来源、
状态持久化断点恢复、关键节点人工审阅。
团队已有 LangChain 经验，未来要做多 Agent 扩展。

## 决策
采用 LangGraph 作为编排框架。

## 备选方案
- CrewAI：角色分工直观，但条件循环/并行汇聚表达弱，HITL 支持不如 LangGraph
- 手写 Loop：最灵活，但要自己实现 checkpoint/重放/HITL，工作量大且易错
- AutoGen：对话式协作好，但研究 Agent 是结构化流程非对话，不匹配

## 后果
好处：分支/循环/并行/HITL/checkpoint 内建；状态可重放调试
代价：学习曲线陡；简单场景过度封装（L10-06 反模式风险）
缓解：节点保持纯函数、recursion_limit 显式、为简单子任务退回手写
```

**ADR 的价值**：
- **可追溯**：半年后查"为啥这么选"，看 ADR 即知
- **防重审**：决策理由记了，避免反复推翻重做
- **可纠错**：后果里写清代价，代价真发生时知道是预期内还是新问题
- **团队对齐**：新人看 ADR 就懂历史决策，不用靠口口相传

### Agent 架构的核心 Trade-off 维度

决策本质是 trade-off——没有完美方案，只有"牺牲 A 换 B"。Agent 架构有四个核心权衡维度：

```
1. 成本（Cost）
   · Token 消耗、模型调用费、基础设施
   · 谁便宜：小模型、缓存、批处理

2. 延迟（Latency）
   · 响应时间、首字延迟、多步累积
   · 谁快：流式、小模型、并行、缓存

3. 可靠性（Reliability）
   · 准确率、稳定性、故障恢复
   · 谁稳：强模型、护栏、重试降级、评估门禁

4. 可维护性（Maintainability）
   · 代码清晰、可调试、团队可接手
   · 谁好维护：抽象合理、文档全、不过度封装
```

**关键认知**：这四个维度**互相矛盾**。要极快可能要牺牲准确（用弱模型）；要极准可能要贵（强模型+多轮）；要极灵活可能难维护。**架构决策就是在四维里找当前最该满足的优先级**。

### Trade-off 量化分析

不能只说"LangGraph 灵活但复杂"——要**量化**。给每个方案在每个维度打分，加权算总分：

```markdown
## Trade-off 分析：编排框架选型

| 方案 | 成本 | 延迟 | 可靠性 | 可维护性 | 加权总分 |
|------|------|------|--------|----------|----------|
| LangGraph | 3 | 3 | 5 | 3 | 3.8 |
| CrewAI | 4 | 4 | 3 | 4 | 3.6 |
| 手写 | 5 | 5 | 3 | 2 | 3.4 |
| AutoGen | 3 | 2 | 3 | 3 | 2.8 |

权重（本项目）：可靠性 0.4 / 可维护性 0.3 / 成本 0.2 / 延迟 0.1
（研究 Agent 质量优先，慢点贵点可接受）

打分依据：
- LangGraph 可靠性 5：内建 checkpoint/HITL，故障可恢复
- 手写可维护性 2：要自实现持久化/HITL，复杂且易错
- ...
```

**量化分析的价值**：
- 把"我觉得 LangGraph 好"变成"在 0.4 可靠性权重下，LangGraph 总分最高"
- 权重显式——不同项目权重不同（客服延迟优先、研究质量优先），决策自然不同
- 打分依据写清——别人能质疑、能调整

> 反模式：**决策只靠"直觉"或"跟风"**。"别人都用 LangGraph 我们也用"——没分析你的场景权重，可能选错。量化分析逼你把假设摆上台面，哪怕打分主观，也比无依据强。

### 常见 Agent 架构决策树

把全书涉及的关键架构选择，凝练成决策树——遇到具体场景能快速定位：

```
决策1：单 Agent 还是多 Agent？
├─ 任务可一人完成、无角色对立 → 单 Agent
└─ 有角色对立/可并行子任务 → 多 Agent（但控规模 2-4）

决策2：手写还是上框架？
├─ 20行能搞定/控制流简单 → 手写
├─ 有分支/循环/并行/HITL/持久化 → 框架（LangGraph）
└─ 角色天然分工 → CrewAI

决策3：RAG 还是微调？
├─ 动态知识、大文档库 → RAG
├─ 固定领域、风格定制 → 微调
└─ 小文档 → 长上下文直接塞

决策4：本地还是云端沙箱？
├─ 数据不能出境/低频 → 本地 Docker
├─ 高频/强隔离/不想运维 → 云端
└─ 完全不可信代码 → 微VM

决策5：长期记忆用哪种？
├─ 跨会话记住偏好 → Mem0 式语义记忆
├─ 超长任务翻历史 → MemGPT 式分层
└─ 单会话 → 短期窗口够

决策6：评估用 LLM-Judge 还是人工？
├─ 大规模自动评测 → LLM-Judge（配人工抽检）
└─ 关键决策/小样本 → 人工
```

**决策树的价值**：把零散的"根据情况选"结构化。新人查决策树能快速定位，不用把全书重读一遍。P14 会要求你为自己的场景走一遍决策树并产出 ADR。

### 决策的常见陷阱

架构决策也有反模式：

```
陷阱1：过早优化
  · 还没用户就为"百万并发"上 K8s + 微服务
  · 对策：先单体跑通，量来了再拆（YAGNI）

陷阱2：追新
  · 新框架/新模型出了就换，不管是否需要
  · 对策：换之前写 ADR，量化新方案的收益是否值得迁移成本

陷阱3：复制大厂
  · "Google/Anthropic 这么做我们也这么做"
  · 对策：大厂的约束（规模/团队/合规）和你不同，抄架构不抄约束

陷阱4：只看技术不看组织
  · 选了"最优"方案但团队不会用 / 招不到人
  · 对策：可维护性维度要算团队能力，技术最优但团队接不住=不可维护

陷阱5：不写 ADR
  · 决策全凭会议讨论，没记录
  · 对策：每个重要决策落 ADR，哪怕简短
```

> 尤其陷阱4——Agent 架构选型常见"选了 LangGraph 因为它最强"，但团队没人会、招���难，最后可维护性极差。**技术决策要算组织约束**，这是架构师区别于技术发烧友的关键。

### 决策是可逆的：降低切换成本

好的架构决策不只选对方案，还**留退路**——万一选错能换。降低切换成本的手段：

```
1. 抽象层隔离
   · 业务逻辑和框架解耦（L10-06）→ 换框架只改编排层
   · LLM 调用封装统一接口 → 换模型只改配置

2. 配置化
   · 模型、阈值、工具集做成配置 → 改配置而非改代码

3. 渐进迁移
   · 新方案和旧方案并行（灰度，M15），量大了再全切

4. 避免深度绑定
   · 别用框架的非标准特性 → 升级/迁移时这些先崩
```

**核心原则**：**架构决策不是一锤定音**。用 ADR 记录"当时这么选的理由 + 预期后果"，事后若约束变了（用户量涨了、团队变了、模型升级了），可以基于新约束重做决策。**ADR 是活文档，约束变了要新增 ADR 修订旧决策**。

### 要点总结

- 架构师核心能力是"会决策"——权衡多个方案选当前最优并记录理由，不是"能实现"
- ADR 五结构：状态/背景/决策/备选方案/后果——让决策可追溯、防重审、可纠错、团队对齐
- 四个核心 trade-off 维度：成本/延迟/可靠性/可维护性——互相矛盾，按场景定权重优先级
- Trade-off 量化：每方案每维度打分，加权算总分，权重显式（客服延迟优先/研究质量优先）
- 决策树：单/多Agent、手写/框架、RAG/微调、本地/云端沙箱、记忆类型、评估方式——结构化定位
- 陷阱：过早优化、追新、复制大厂（抄架构不抄约束）、只看技术不看组织、不写ADR
- 决策可逆：用抽象层/配置化/渐进迁移/避免深度绑定降低切换成本
- ADR 是活文档：约束变了（量/团队/模型）基于新约束重做决策，新增 ADR 修订旧决策
