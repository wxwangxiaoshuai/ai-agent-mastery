## 架构演进、版本兼容与治理

前面几节讲"怎么设计"。但 Agent 系统不是一次性交付——它会长期演进：模型升级、Prompt 改、工具加、需求变。这一节收口 M14：**怎么让系统在持续演进中不崩**——版本管理、向后兼容、灰度迁移、数据治理与合规。这是架构师最容易忽视、出事就大的领域。

### 演进是常态，不是例外

先建立正确认知——Agent 系统的"不变"是例外，"变"才是常态：

```
会变的东西：
  · 模型：GPT-4o → 下一代、换 Claude、版本升级
  · Prompt：改一个字词、调结构、加规则
  · 工具：加新工具、改老工具签名、废弃工具
  · 记忆 schema：加字段、改结构
  · 业务需求：新场景、新约束、新合规
  · 依赖库：LangGraph 升级、SDK 变 API

不变的假设是危险的：
  · "Prompt 不会改" → 改了发现全链路依赖它格式
  · "模型不会换" → 换了发现输出格式变了全崩
  · "记忆结构固定" → 加字段发现老数据不兼容
```

**架构师的演进思维**：设计时就**假设它会变**，并为此留好机制——版本化、兼容层、灰度、治理。不是等变了再手忙脚乱。

### 三维版本管理：Prompt、工具、模型

Agent 系统的版本复杂在它是**三维的**，不像普通软件只是一个代码版本：

```
普通软件版本：v1.2.3（代码版本）
Agent 系统三维版本：
  1. Prompt 版本：system prompt 改了 → 行为变
  2. 工具版本：工具签名/行为变了 → 兼容性
  3. 模型版本：换模型/模型升级 → 输出特性变

三者独立演进，但互相影响：
  · Prompt v5 + 模型 v2 + 工具 v3 = 一套配置
  · 改任一维都可能破坏配合
```

**版本管理的工程化**：

```python
# 配置化 + 版本标记
AGENT_CONFIG = {
    "version": "2026.07.23",   # 整体配置版本
    "prompt": {"id": "sys_v5", "source": "prompts/sys_v5.md"},
    "model": {"name": "gpt-4o-2026-07", "fallback": "gpt-4o-mini"},
    "tools": ["search@v3", "code_exec@v2", "fetch@v1"],   # 工具带版本
}

# 每次请求记录用了哪套版本（关联 trace，L13-03）
def run(agent_config):
    with trace.span("agent_run"):
        trace.set_attribute("config.version", agent_config["version"])
        trace.set_attribute("prompt.id", agent_config["prompt"]["id"])
        # ... 这样出问题能定位是哪套版本
```

**三维独立版本化的价值**：
- Prompt 改了不影响工具/模型版本 → 能独立回归测试
- 出问题能定位"是 prompt 改的、模型升级的、还是工具变的"
- 回滚能精确到某维（只回滚 prompt，不回滚工具）

### 向后兼容：别让老用户/老数据崩

演进最大的坑——**新版本破坏老用法/老数据**：

```
不兼容的灾难：
  · Prompt v6 改了输出格式 → 依赖 v5 格式的下游全崩
  · 工具 search@v4 改了返回字段 → 用 v3 字段的代码全错
  · 记忆 schema 加字段 → 老记忆没这字段，加载崩
  · 模型换了 → 以前能结构化输出的现在不输出了
```

**兼容的三类对象**：

```
1. 用户的会话/历史：老对话在新版本要能继续
   · 对策：对话历史格式带版本，新版本能解析老格式

2. 存储的记忆/数据：老记忆在新 schema 下要可用
   · 对策：schema 演进用"加字段不删字段"+ 默认值

3. API/工具契约：老调用方在新工具下要能工作
   · 对策：工具签名兼容（加参数给默认值、不删老参数）
```

```python
# 记忆 schema 演进的兼容原则
class Memory:
    # v1: {user_id, fact}
    # v2: 加 type 字段
    # v3: 加 created_at 字段
    
    def load(self, record: dict):
        # 兼容老数据：新字段没有给默认值
        return Memory(
            user_id=record["user_id"],       # 必填，老数据有
            fact=record["fact"],             # 必填，老数据有
            type=record.get("type", "fact"), # v2加，老数据默认"fact"
            created_at=record.get("created_at", ""),  # v3加，老数据默认空
        )
```

**兼容原则**：
- **加字段不删字段**（加默认值兼容老数据）
- **改行为不改契约**（老 API 签名不变，内部实现可改）
- **删功能先标记废弃**（deprecation warning），给迁移期，再删

### 灰度迁移：别一刀切

大改动（换模型、改 prompt 架构）不能一刀切全量上——**灰度发布**逐步切：

```
灰度迁移流程：
  1. 影子流量：新版本跑真实流量但不返回用户（只对比）
     · 看新旧版本输出差异，发现回归
  2. 小流量灰度：5% 用户走新版本
     · 监控质量指标，没问题逐步放量
  3. 扩大灰度：20% → 50% → 80%
     · 每步看指标，有问题立即回滚
  4. 全量 + 保留回滚能力
```

```python
def route_by_canary(user_id, new_version_ratio=0.05):
    """灰度路由：按用户 hash 分配版本"""
    # 用 hash 而非随机 → 同一用户稳定在同一版本（避免来回切）
    h = hash(user_id) % 100
    return "new" if h < new_version_ratio * 100 else "old"

def agent_run(user_id, question):
    version = route_by_canary(user_id)
    config = NEW_CONFIG if version == "new" else OLD_CONFIG
    result = run_with(config, question)
    # 对比新旧（影子流量时）
    if config == OLD_CONFIG and shadow_enabled:
        new_result = run_with(NEW_CONFIG, question)  # 跑新不返回
        log_diff(result, new_result)   # 记录差异分析
    return result
```

**灰度的关键**：
- **稳定分流**：同一用户稳定在一版（用 hash 不用随机），否则来回切体验崩
- **可回滚**：任何阶段发现问题，立即把流量切回老版本
- **指标对比**：新旧版本的质量/成本/延迟指标对比，L13-02 评测流水线在这里用

> 灰度是 L13-02 评测、L13-03 trace 的综合应用——上线前评测把质量门、灰度时 trace 对比指标、出问题靠 trace 定位。前几节的质量保障体系在演进时全用上。

### 数据治理与合规审计

Agent 系统处理大量数据（用户对话、记忆、文档），治理和合规是硬约束：

```
数据治理维度：
  1. 数据留存策略：什么数据存多久
     · 对话历史存多久（合规/成本权衡）
     · 记忆过期清理（M8 遗忘机制）
  
  2. 数据访问审计：谁访问了什么
     · 审计日志（L13-06）记录数据访问
     · 敏感数据访问要授权
  
  3. 数据主权：用户对数据有控制权
     · 导出：用户能拿走自己的数据
     · 删除：用户能要求删自己的数据（GDPR 被遗忘权）
     · 这要求"按用户/租户能精确删除"
  
  4. 数据出境：数据不能随便跨境
     · 多租户平台尤其要注意（L14-05）
     · 云端沙箱/LM 调用都涉及数据出境
```

**合规审计框架**：

```python
class ComplianceAudit:
    """合规审计框架"""
    def __init__(self):
        self.access_logs = []   # 数据访问日志
        self.data_inventory = {}  # 数据清单
    
    def log_access(self, accessor, data_type, data_owner, purpose):
        """记录数据访问"""
        self.access_logs.append({
            "timestamp": now(), "accessor": accessor,
            "data_type": data_type, "owner": data_owner,
            "purpose": purpose,   # 访问目的（合规要）
        })
    
    def export_user_data(self, user_id):
        """用户数据导出（数据主权）"""
        return collect_all_user_data(user_id)
    
    def delete_user_data(self, user_id):
        """用户数据删除（被遗忘权）"""
        # 从所有存储精确删除该用户数据
        delete_from_vector_db(user_id)
        delete_from_history(user_id)
        delete_from_memory(user_id)
        log_deletion(user_id)   # 删除也要留审计
```

**关键合规要求**：
- **可审计**：能证明"某时某数据被谁访问、做什么用"
- **可删除**：用户要求删，能精确删干净（不留残骸）
- **最小化**：只收集必要数据，不过度留存

### 演进路线图：有计划地变

好的演进不是"出了问题才改"，是**有路线图地主动演进**：

```
演进路线图示例（企业客服 Agent）：
  Q1：单 Agent MVP，基础对话+知识库
  Q2：加多轮记忆、加评估流水线（L13-02）
  Q3：平台化（多租户，L14-05）、加护栏（L13-04）
  Q4：多 Agent 协作（M11）、AaaS 化

每个里程碑：
  · 明确要解决的架构问题
  · 对应的 ADR（L14-01）
  · 风险与回滚预案
  · 成功指标（质量/规模/成本）
```

**路线图的价值**：
- **预期管理**：团队和利益相关者知道下一步做啥
- **防冒进**：不一次做太多，分里程碑验证
- **资源规划**：提前知道要扩什么（L14-05 平台化要扩团队）

### 演进的反模式

```
反模式1：大爆炸重写
  · "推倒重来做一个新 Agent" → 旧的问题没解决、新的问题一堆
  · 对策：渐进演进，Strangler Fig 模式（新功能逐步替换老系统）

反模式2：不写 ADR 演进
  · 每次改都没记录 → 半年后不知道为啥变成现在这样
  · 对策：每次架构改动落 ADR（L14-01）

反模式3：无灰度全量上
  · 新 prompt 直接全量 → 质量崩了影响所有用户
  · 对策：灰度（这节）

反模式4：只加不删/不弃
  · 老工具/老字段/老 prompt 一直留着 → 系统臃肿、维护成本涨
  · 对策：标记废弃 → 给迁移期 → 删除（这节兼容原则）
```

### 要点总结

- 演进是常态不是例外：模型/Prompt/工具/记忆 schema/需求都会变，设计时假设会变并留机制
- 三维版本管理：Prompt + 工具 + 模型独立版本化，整体 config 带 version，每次请求记录版本关联 trace
- 向后兼容三对象：会话历史、记忆数据、API契约；原则=加字段不删字段(给默认值)+改行为不改契约+删功能先标记废弃
- 灰度迁移：影子流量对比→小流量→扩大→全量；稳定分流(hash非随机)+可回滚+指标对比(L13-02/03综合应用)
- 数据治理四维：留存策略、访问审计、数据主权(导出/删除被遗忘权)、数据出境
- 合规审计：可审计(证明访问)、可删除(精确删干净)、最小化(只收必要)
- 演进路线图：有计划主动变，分里程碑、每步配 ADR+风险+指标
- 反模式：大爆炸重写(用Strangler Fig渐进)、不写ADR、无灰度全量、只加不删
- M14 收官：能力地图 能实现→能决策→能拆解→能设计平台→能驾驭演进；P14 产出真实架构文档完成跨越
