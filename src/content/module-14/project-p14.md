## Agent 架构设计文档

M14 六节课讲了架构决策方法论、业界案例拆解（Copilot/Cursor/Devin/Claude Code/Perplexity/ChatGPT）、平台架构、演进治理。P14 不是写代码——而是产出一份**完整的 Agent 架构设计文档**。这是从"能实现"到"会设计"的最终跨越：选一个真实业务场景，做架构决策、画图、量化 trade-off、估容量、定安全方案、画演进路线，并对标业界参考架构。

### 项目目标

选择一个真实业务场景，产出完整架构设计文档：
- 场景选型（企业智能客服 / 研发助手 / 数据分析 Agent / 自选）
- 架构决策记录（ADR）至少 5 条
- 系统架构图与数据流图
- Trade-off 量化分析报告
- 容量估算与成本模型
- 安全方案与合规策略
- 架构演进路线图
- 对标至少一个业界参考架构

### 验收标准

- [ ] 选定真实业务场景，明确产品定位与自主性谱系位置（L14-01/03）
- [ ] ADR ≥ 5 条，每条含状态/背景/决策/备选方案/后果（L14-01）
- [ ] 系统架构图 + 数据流图，清晰展示分层与组件
- [ ] Trade-off 量化分析：四维度（成本/延迟/可靠性/可维护性）打分 + 权重
- [ ] 容量估算：并发/存储/带宽/成本预测
- [ ] 安全方案：数据隔离/护栏/沙箱/审计/合规
- [ ] 演进路线图：分里程碑，每步配 ADR+风险+指标
- [ ] 对标业界参考架构（如 Perplexity 搜索型 / Cursor 编程型 / 某客服平台）
- [ ] 文档自洽：新人能据此理解系统为何这么设计

### 示例场景：企业智能客服 Agent

为帮你起步，给一个示例场景（你可自选其他）。**企业智能客服**——替代传统客服，能理解用户问题、查知识库、办理业务、必要时转人工。

**产品定位**（L14-01/04）：
- 自主性谱系：半自主（Agent 处理常见问题，复杂/高风险转人工，L14-03）
- 架构哲学：对话型 + 搜索型混合（类 Perplexity 答案引擎 + ChatGPT 对话，L14-04）
- 关键约束：延迟敏感（客服不能让用户等 30s）、准确率优先（答错比慢更糟）、合规严（用户数据/对话留存）

### Step 1：ADR 示例（≥5 条）

```markdown
# ADR-001：编排框架选用 LangGraph

## 状态
已接受

## 背景
客服 Agent 需要：多轮对话状态管理、条件分支（简单答/转人工/办业务）、
对话历史 checkpoint（断网恢复）、关键操作 HITL（办业务要确认）。

## 决策
采用 LangGraph 编排主流程。

## 备选方案
- CrewAI：角色分工好，但客服是结构化控制流非角色协作，HITL 支持弱
- 手写 Loop：灵活但 checkpoint/HITL 要自实现，客服量大不容错

## 后果
好处：HITL/checkpoint/分支内建；状态可重放调试（L13-03）
代价：学习曲线；简单问答过度封装风险
缓解：节点纯函数、recursion_limit 显式、简单子任务退回手写
```

```markdown
# ADR-002：知识库用 RAG（非微调）

## 状态
已接受

## 背景
客服知识（产品手册/政策/FAQ）频繁更新，且要可追溯来源。

## 决策
用 RAG（M4）+ 引用溯源，不微调模型。

## 备选方案
- 微调：知识固化进模型，但更新要重新训练、无法溯源
- 长上下文：全量塞，但知识库大、成本高、Lost in Middle

## 后果
好处：知识实时更新、答案带引用可验证、更新便宜
代价：检索质量依赖分块/Embedding；要做 RAGAS 评估（M4）
缓解：混合检索+Reranking、RAGAS 评测流水线（L13-02）

（ADR-003 记忆架构 Mem0 式、ADR-004 多租户隔离策略、ADR-005 模型路由 Opus→Sonnet→Haiku ...）
```

### Step 3：数据流图（关键场景）

```
场景：用户问"怎么退订我的订单"

用户消息
  │
  ├→ 输入护栏：查 PII/注入 → 通过
  ├→ 意图判断：办业务（退订）
  ├→ 记忆 recall：该用户历史订单/偏好
  ├→ 知识库 RAG：退订政策
  ├→ 工具：查订单系统 → 该用户有订单 #123
  ├→ HITL：退订是要钱的操作 → 暂停等确认（L10-03）
  │    （用户确认）→ 执行退订 → 工单系统
  ├→ 输出护栏：事实校验（退订政策对不对）
  └→ 返回用户 + 记录 trace + 审计日志
```

### Step 4：Trade-off 量化分析

```markdown
## 四维度打分（客服场景，权重：可靠性0.4/延迟0.3/成本0.2/可维护0.1）

| 方案 | 可靠性 | 延迟 | 成本 | 可维护 | 加权总分 |
|------|--------|------|------|--------|----------|
| 强模型(4o)+RAG+HITL | 5 | 3 | 2 | 3 | 3.9 |
| 弱模型(mini)+RAG | 3 | 5 | 5 | 4 | 3.7 |
| 微调+无RAG | 3 | 4 | 2 | 2 | 3.0 |

决策��选强模型+RAG+HITL——客服质量优先（答错成本最高），
延迟通过流式+模型路由（简单问题用mini，复杂用4o）缓解。
```

### Step 5：容量与成本模型

```markdown
## 容量估算
· 峰值并发：预估 1000 并发会话
· 每会话平均：5 轮 × 1500 token = 7500 token
· 峰值 token/秒：1000 × 7500 / 300s ≈ 25000 token/s

## 成本预测（月）
· LLM：假设 60% 走 mini(便宜) + 40% 走 4o
  · 月调用量估算 → token 成本
· 向量库：按租户分片，存储成本
· 基础设施：无状态 Agent 实例 × 扩容数

## 容量风险
· 峰值超 25k token/s 可能触发限流 → 多模型供应商分流（M7）
· 知识库增长 → 检索延迟涨 → 分片 + 缓存
```

### Step 6：安全与合规方案

```markdown
## 安全方案
· 多租户隔离：tenant_id 强制过滤，封装租户感知查询（L14-05）
· 输入护栏：PII 检测脱敏、注入检测（L13-04/05）
· 输出护栏：退订政策事实校验、不泄露他人订单（L13-06 所有权约束）
· HITL：办业务/退款前人工确认（L10-03）
· 沙箱：业务工具执行隔离（M9）
· 审计：所有业务操作留审计日志（L13-06）

## 合规
· 对话留存策略：保留 N 天供质检，过期清理
· 数据主权：用户可导出/删除自己的对话
· 数据出境：多租户数据不出境（本地化部署/私有云）
```

### Step 7：演进路线图

```markdown
## 路线图
M1-M3：单 Agent MVP（对话+RAG+基础护栏）
M4-M6：加记忆、加评估流水线（L13-02）、加全链路 trace（L13-03）

```

### Step 8：业界对标

```markdown
## 对标 Perplexity（L14-04）
· 相似：答案带引用溯源、知识库 RAG
· 差异：客服是对话型+办业务，Perplexity 是搜索型纯问答
· 借鉴：引用溯源、查询改写、多查询并行

## 对标 ChatGPT（L14-04）
· 相似：多轮对话、长期记忆、工具按需
· 差异：客服有强合规/隔离约束，工具是业务系统非通用
· 借鉴：记忆架构、工具决策、模型路由

## 对标 Cursor/Claude Code（L14-02/03）
· 借鉴：HITL 在关键操作暂停（Claude Code 的写操作确认）
· 差异：客服不自主编程，自主性低一档
```

### 实施步骤总结

1. **选场景**：选一个你熟悉/感兴趣的真实业务，明确产品定位
2. **定谱系**：在自主性谱系定位（辅助/半自主/高自主），定架构重点
3. **写 ADR**：≥5 条核心架构决策，每条五结构
4. **画架构图**：分层 + 组件 + 数据流
5. **量化分析**：四维度打分 + 权重，论证关键选择
6. **估容量成本**：并发/存储/成本预测，找瓶颈
7. **定安全合规**：隔离/护栏/沙箱/审计/合规
8. **画路线图**：分里程碑，每步配 ADR+风险+指标
9. **对标业界**：至少一个参考架构，说借鉴与差异
10. **风险与未决**：诚实列还没解决的问题

### 进阶挑战

1. **多场景对比**：同一架构给"客服/研发助手/数据分析"三个场景各做一份 ADR，对比权衡差异
2. **容量压测设计**：基于容量模型设计压测方案，验证瓶颈假设
3. **成本优化方案**：在成本模型上设计优化（模型路由/缓存/蒸馏，M15）
4. **治理框架细化**：把合规审计框架细化到可执行（数据清单/访问授权流程）
5. **对标更多案例**：除业界产品外，对标开源 Agent 项目架构

### 要点回顾

- P14 是"产出架构文档"而非写代码——从能实现到会设计的最终跨越
- 文档结构：执行摘要/背景目标/ADR/架构图/数据流/Trade-off/容量成本/安全合规/路线图/对标/风险
- ADR ≥5 条：每条状态/背景/决策/备选方案/后果（L14-01）
- 架构图分层：接入/编排/知识/记忆/工具/质量安全/平台，对应 M3-M13 各模块
- Trade-off 量化：四维度打分+权重，权重按场景（客服质量优先）
- 容量成本：并发/token/s/存储预测，找瓶颈（限流/检索延迟）
- 安全合规：多租户隔离+输入输出护栏+HITL+沙箱+审计+留存/主权/出境
- 演进路线图：分里程碑，每步 ADR+风险+指标（L14-06）
- 业界对标：学权衡思路不抄实现，说借鉴与差异
- 诚实列风险与未决项——架构文档不是吹嘘，是工程决策的诚实记录
