## 参考架构案例拆解 I：Cursor 与 GitHub Copilot

L14-01 讲了决策方法论。这一节开始用它拆真实产品——**AI 编程助手**是 Agent 最成熟的落地领域，Cursor 和 GitHub Copilot 是标杆。拆解它们不是抄代码，是学**架构决策的实战演绎**：在产品约束下，怎么取舍。这一节拆 Copilot 的补全和 Cursor 的 Agent 模式。

### 为什么先拆编程助手

AI 编程助手有几个特点让它特别适合做架构教材：

```
· 成熟：商业化最早、迭代最久、模式沉淀最清晰
· 可观察：公开行为多（补全/编辑/多轮），能反推架构
· 约束鲜明：延迟极敏感（补全要毫秒级）、上下文极复杂（整个 codebase）
· 技术全栈：涉及上下文工程、RAG（codebase索引）、工具（文件操作）、Agent Loop
```

> 学这节的目标：把前面学的上下文工程(M3)、RAG(M4)、工具(M6)、Agent Loop(M5)、沙箱(M9) 在一个真实产品里"对号入座"。你会看到这些技术不是孤立的，是组合成一个产品的。

### GitHub Copilot：补全型架构

Copilot 最初的核心能力是**代码补全**——你写到一半，它猜你接下来写啥。架构核心是**上下文组装 + 低延迟补全**：

```
Copilot 补全架构（简化）：
  编辑器 ──→ 上下文抽取 ──→ 补全请求 ──→ 模型 ──→ 后处理 ──→ 插回编辑器
              ↑                                              ↓
              └──────── 用户接受/拒绝反馈 ←─────────────────┘

关键环节：
  1. 上下文抽取：当前文件 + 光标位置 + 少量相关文件
  2. 低延迟模型：用快/小模型（不是最大模型），补全要 <300ms
  3. 后处理：语法校验、格式化、去重复
  4. 反馈学习：接受/拒绝信号回流（早期靠此优化）
```

**架构决策要点（用 L14-01 的视角看）**：

| 决策 | Copilot 的选择 | 理由（trade-off） |
|------|----------------|-------------------|
| 模型 | 小/快模型而非最大 | 补全延迟敏感（<300ms），准确率让步于速度 |
| 上下文范围 | 当前文件为主 + 极少量跨文件 | 上下文越长延迟越高、成本越高；补全主要靠局部上下文 |
| 触发方式 | 被动触发（用户停顿） | 主动补全会打扰，被动等停顿判断意图 |
| 输出长度 | 短（几行内） | 长补全准确率降、用户难审；短补全快、易接受 |

**核心架构教训**：
- **补全型 Agent 牺牲质量换延迟**——不能用最大模型，因为延迟是体验生命线。这印证 L14-01 的"延迟/可靠性 trade-off"。
- **上下文范围要克制**——不是越多越好，多上下文 = 多延迟 + 多成本。补全靠局部，深度理解靠别的机制（Cursor 的 codebase 索引）。

### Copilot 的演进：从补全到 Chat 到 Agent

Copilot 不是停在补全，它在演进——这个演进路径本身就是架构演进教材：

```
阶段1：补全（被动补全代码）
  · 架构：上下文抽取 + 快模型 + 后处理

阶段2：Chat（对话问答）
  · 架构：加对话管理 + 上下文检索（@workspace 跨文件）
  · 新问题：用户问题可能跨文件 → 需要轻量 RAG 检索相关代码

阶段3：Agent / Edit（主动改代码）
  · 架构：Agent Loop + 工具（读/写文件、运行命令）
  · 新问题：多步编辑要可靠、改动要可审可回滚
```

**演进启示**：每个阶段叠加新能力，但**前一阶段的架构不能被推翻**——补全还在用，只是多了 Chat/Agent 模式。这是 L14-06 要讲的"向后兼容"——架构演进是叠加，不是推倒重来。

### Cursor：Agent 模式架构

Cursor 是"原生 Agent"的编程助手——不是补全为主，而是**Agent 主动完成多步编码任务**。架构核心是 **Codebase 索引 + 工具链 + 多轮编辑**：

```
Cursor Agent 架构（简化）：
  用户："给这个函数加错误处理"
    │
    ├─→ 1. Codebase 索引：embedding 索引整个仓库
    │      · 离线构建 + 增量更新
    │      · 检索相关文件/符号
    │
    ├─→ 2. 上下文组装：检索结果 + 当前文件 + 用户指令
    │
    ├─→ 3. Agent Loop（M5）：
    │      Thought → Action(读文件/改文件/跑测试) → Observation → ...
    │
    ├─→ 4. 多轮编辑：逐个文件改，带 diff 预览
    │
    └─→ 5. 验证：跑测试/lint，失败则修复循环
```

### Cursor 的关键架构决策

逐个看 Cursor 的架构选择，都在印证前面学过的技术：

**决策1：Codebase 索引用 RAG（M4）**
- 整个仓库太大塞不进上下文 → 用 embedding 索引，按需检索相关代码
- **离线索引 + 增量更新**：仓库变就更新索引，而非每次查询重建
- 检索的不是"文档块"而是"代码符号/函数"——代码的 chunking 要按语法结构（M4 分块策略）

**决策2：Agent Loop 而非一次性生成（M5）**
- 复杂改动一次生成不可靠 → Agent 多步：读→改→验证→修
- 每步有 diff 预览，用户可审可控（L10-03 HITL 思想）

**决策3：工具链设计（M6）**
- 工具：读文件、写文件、grep、跑命令、跑测试
- **工具粒度**：不过粗（"改代码"太大）也不过细（"插入字符"太小）——文件级操作 + 命令执行

**决策4：沙箱执行（M9）**
- 跑测试/命令要安全 → 沙箱或受限环境
- 用户可见执行过程，可中断

**决策5：多轮编辑的可靠性（M7）**
- 多步改动可能失败 → checkpoint 状态、可回滚
- 测试失败 → 自动进入修复循环

### Cursor vs Copilot：架构权衡对比

两者都做"AI 编程"，但架构哲学不同：

| 维度 | Copilot（补全为主） | Cursor（Agent 为主） |
|------|---------------------|----------------------|
| 核心能力 | 被动补全 | 主动多步编辑 |
| 延迟要求 | 极高（毫秒级首字） | 容忍秒级（Agent 多步） |
| 模型选择 | 小/快模型 | 强模型（多步要准） |
| 上下文 | 当前文件为主 | Codebase 全局索引 |
| 用户控制 | 接受/拒绝补全 | 审 diff/可中断/可回滚 |
| 可靠性重点 | 补全准确率 | 多步编辑的可控可回滚 |
| 适合 | 写代码时辅助 | 完成明确任务 |

**关键洞察**：
- **Copilot 优化"延迟+局部"**，Cursor 优化"全局+可控"——不同产品定位导致完全不同的架构权衡
- **不是 Cursor 比 Copilot 先进**，是解决不同问题：Copilot 辅助你写（人主导），Cursor 替你写（Agent 主导）。二者可以共存（事实上现在互相借鉴）

> 这印证 L14-01 的核心：**没有"更好"的架构，只有"匹配产品定位"的架构**。Copilot 的延迟敏感让它不能用 Cursor 的强模型慢架构；Cursor 的多步任务让它不能用 Copilot 的一次性快补全。架构服务于产品定位。

### 提炼：AI 编程助手的通用架构模式

拆完两个产品，提炼共通的架构模式：

```
AI 编程助手通用架构：
  1. Codebase 索引层（RAG）
     · embedding 索引 + 语法感知分块
     · 离线构建 + 增量更新
  2. 上下文组装层（M3 上下文工程）
     · 当前上下文 + 检索结果 + 用户指令
     · Token 预算管理
  3. 推理/执行层（Agent Loop / 补全）
     · 补全型：快模型 + 一次性
     · Agent型：强模型 + 多步 + 工具
  4. 工具层（M6）
     · 文件读写、命令执行、搜索
  5. 安全/沙箱层（M9）
     · 代码执行隔离、可回滚
  6. 反馈层
     · 接受/拒绝、diff 审查、中断
```

**这些模式对应你学过的**：RAG(M4)、上下文工程(M3)、Agent Loop(M5)、工具(M6)、沙箱(M9)、HITL(L10-03)。**全书技术在一个产品里全用上了**——这是为什么编程助手是好教材。

### 拆解的边界：别把推测当事实

重要提醒——**这些都是基于公开行为和常识的架构推测，不是官方实现**：

```
拆解的可信度：
  · 公开行为（补全/编辑/索引）→ 可靠推断
  · 内部模型选择、具体优化 → 推测，未必准
  · 产品在迭代 → 今天对明天可能变

正确的态度：
  · 学架构模式和权衡思路，而非当"真相"
  · 关注"为什么这么设计"（可迁移）而非"具体怎么实现"（会变）
```

> 这节和后续 L14-03/04 都是"学思路不抄实现"。业界案例的价值是让你看到真实约束下的真实决策，帮你建立架构直觉——不是让你照搬。P14 会让你的架构文档"对标业界参考架构"，就是用这种思路。

### 要点总结

- AI 编程助手是 Agent 最成熟落地，拆它学架构权衡最值——技术全栈（上下文/RAG/工具/Loop/沙箱）在一个产品里对号入座
- Copilot 补全架构：上下文抽取+快模型+后处理+反馈；核心权衡是牺牲质量换延迟（小模型<300ms）
- Copilot 演进：补全→Chat→Agent，每阶段叠加能力不推翻前架构（L14-06 向后兼容）
- Cursor Agent 架构：Codebase 索引(RAG)+工具链+多轮编辑+沙箱+可靠性(可回滚)
- Cursor 关键决策：RAG索引代码按语法分块、Agent Loop多步、工具文件级、沙箱执行、多轮可回滚
- Copilot vs Cursor：补全型优化延迟+局部，Agent型优化全局+可控——不同定位不同权衡，无高下
- 通用模式六层：索引层/上下文层/推理层/工具层/安全层/反馈层——对应M3/M4/M5/M6/M9/L10-03
- 核心教训：没有"更好"架构，只有匹配产品定位的——Copilot延迟敏感不能慢，Cursor多步不能一次性
- 拆解边界：学思路不抄实现，公开行为可靠、内部细节是推测，产品在迭代
