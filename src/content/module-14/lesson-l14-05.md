## Agent 平台架构：多租户、可扩展、Agent-as-a-Service

前几节拆的是"单个 Agent 产品"。但当你给企业做 Agent——不是给一个用户做一个 Agent，而是给**多个客户、每个客户多个 Agent**——架构问题完全不同了：怎么让不同客户的数据隔离？怎么水平扩容？怎么把"做 Agent"变成"卖 Agent 服务"？这一节讲 Agent 平台架构。

### 从单 Agent 到 Agent 平台

先理解量变到质变——规模上来了，架构问题就变了：

```
单 Agent / 少量用户：
  · 一个 Agent 服务，少量并发
  · 数据都在一个库
  · 重点：Agent 本身能力（前面 13 模块）

Agent 平台 / 多客户多 Agent：
  · 成百上千客户，每客户多个 Agent
  · 客户数据必须隔离（A 看不到 B 的）
  · 高并发要水平扩展
  · 重点：隔离、扩展、计费、治理（本节 + L14-06）
```

**质变点**：单 Agent 你操心"Agent 聪不聪明"；平台你操心"成百上千个 Agent 怎么管、数据怎么隔、怎么扩、怎么计费"。这是从"做应用"到"做平台"的跨越。

### 多租户：数据隔离是底线

多租户（multi-tenant）= 一套系统服务多个客户（租户），但客户间数据严格隔离。这是 Agent 平台的**安全底线**：

```
多租户隔离失败 = 灾难：
  · 客户 A 的 Agent 检索到客户 B 的私有文档 → 数据泄露
  · 客户 A 的用户记忆被客户 B 检索到 → 隐私事故
  · 商业上：客户绝不会信任会串数据的平台
```

**隔离的三层**：

```
1. 数据隔离：不同租户的数据物理/逻辑隔离
   · 每个查询带 tenant_id 过滤（M8 记忆的用户隔离是这个的延续）
   · 或物理分库（每个租户独立库/独立 schema）

2. 资源隔离：一个租户不能用尽资源拖垮别人
   · 配额：每租户 token/请求数上限
   · 限流：防某租户刷爆 API（M7 限流）

3. 权限隔离：租户内不同角色权限不同
   · 管理员/普通用户，能看的 Agent/数据不同
```

```python
# 数据隔离：所有查询强制带 tenant_id
class TenantScopedMemory:
    def __init__(self, tenant_id: str):
        self.tenant_id = tenant_id   # 租户隔离的核心

    def recall(self, query, user_id):
        # 关键：where 条件同时过滤 tenant_id 和 user_id
        return collection.query(
            query_texts=[query], n_results=5,
            where={"$and": [
                {"tenant_id": self.tenant_id},   # 租户隔离
                {"user_id": user_id},            # 用户隔离
            ]})

    def remember(self, user_id, dialog):
        # 写入时强制带 tenant_id
        facts = extract(dialog)
        collection.add(documents=facts, metadatas=[
            {"tenant_id": self.tenant_id, "user_id": user_id, ...}])
```

> 串数据的根因几乎都是"某个查询忘了带 tenant_id 过滤"。**架构上要保证不可能忘**——封装成统一的租户感知查询接口，禁止裸查向量库。L13-06 的"最小权限+所有权约束"在这里升级为"租户隔离强制"。

### 多租户隔离的三种实现策略

按隔离强度从低到高（成本从低到高）：

```
策略1：逻辑隔离（共享库 + tenant_id 字段）
  · 所有租户数据在一个库，靠 tenant_id 过滤
  · 最省资源、好维护
  · 风险：靠代码保证不漏过滤，一次漏就串
  · 适合：中小租户、信任度高

策略2：Schema 隔离（共享库实例 + 独立 schema）
  · 每租户独立 schema（表结构）
  · 物理上更隔离，查询天然不跨租户
  · 适合：中型租户、要一定隔离

策略3：物理隔离（独立库/独立实例）
  · 每租户独立数据库甚至独立服务实例
  · 最强隔离
  · 成本最高、运维重
  · 适合：大客户、强合规要求（金融/医疗）
```

| 策略 | 隔离强度 | 成本 | 适合 |
|------|---------|------|------|
| 逻辑（tenant_id） | 弱（靠代码） | 低 | 中小租户 |
| Schema | 中 | 中 | 中型租户 |
| 物理 | 强 | 高 | 大客户/合规 |

**混合策略**：平台常**分级**——免费/小客户用逻辑隔离，企业大客户用物理隔离。给钱多的给强隔离，符合商业和合规。

### 可扩展：水平扩展 Agent 服务

用户多了，单机扛不住。怎么水平扩展（scale out）？

```
垂直扩展（scale up）：换更强机器
  · 简单但有上限（单机再强也有天花板）
  · 成本线性涨甚至超线性

水平扩展（scale out）：加更多机器
  · 理论上无限扩展
  · 前提：服务无状态化（stateless）
```

**无状态化是水平扩展的前提**——这是架构上的硬要求：

```
有状态（不能水平扩展）：
  · Agent 把对话历史/记忆存在本机内存
  · 下次请求如果打到另一台机器 → 历史丢了
  · 只能粘性路由（同一用户固定一台），扩展性差

无状态（可水平扩展）：
  · Agent 把状态存外部（DB/Redis/向量库）
  · 每次请求从外部加载状态
  · 任意机器都能处理任意用户 → 负载均衡自由
```

```python
# 无状态 Agent：状态全在外部
class StatelessAgent:
    def run(self, tenant_id, user_id, question):
        # 每次从外部加载状态，不在本机存
        history = load_history_from_db(tenant_id, user_id)   # 外部DB
        memory = TenantScopedMemory(tenant_id).recall(question, user_id)  # 外部向量库
        
        result = self._llm(history, memory, question)
        
        # 状态写回外部，不在本机留
        save_history_to_db(tenant_id, user_id, result)
        return result
```

**无状态化的代价**：每次请求要从外部加载状态，多一次 IO。但换来的是**可自由水平扩展+负载均衡+容错**（某台挂了，请求转其他机器，状态在外部不丢）。这个 trade-off 几乎总是值——生产平台必无状态化。

### 水平扩展的配套

光无状态不够，水平扩展要配套：

```
1. 负载均衡
   · 请求均匀分到多台机器
   · 策略：轮询/最少连接/一致性hash
   · 长任务（Agent 多步）要考虑会话粘性或状态外置

2. 分片（Sharding）
   · 数据量大一台存不下 → 分片到多台
   · 按 tenant_id 分片：某租户数据固定在某台
   · 向量库分片：按租户/按空间分

3. 缓存
   · 语义缓存：相似问题直接返回缓存（M15 详讲）
   · 热点租户/热点 Agent 缓存

4. 异步队列
   · 长任务（深度研究 Agent）异步执行
   · 用户提交→入队→异步处理→通知结果（M7 队列）
```

### Agent-as-a-Service：把 Agent 做成服务

AaaS（Agent-as-a-Service）= 平台不只跑自己的 Agent，还**让客户/开发者构建和部署他们自己的 Agent**。这是平台化的更高形态：

```
AaaS 的能力（类比 SaaS/PaaS）：
  1. Agent 构建：低代码/SDK 让用户定义自己的 Agent
     · 配置 prompt、选工具、接知识库
     · 不用写代码也能搭 Agent
  2. Agent 部署：一键部署到平台
     · 平台管扩容、监控、计费
  3. Agent 市场：共享/买卖 Agent
  4. 工具市场：Agent 可挂载第三方工具
  5. 计费计量：按用量/按租户计费
```

**AaaS 架构组成**：

```
┌─────────────── AaaS 平台 ──────────────────┐
│  控制面（管理）                              │
│   · 租户管理、Agent 配置、计费、监控         │
├────────────────────────────��────────────────┤
│  数据面（运行）                              │
│   · Agent 运行时（无状态、可扩展）           │
│   · 工具执行沙箱（M9）                       │
│   · 知识库/记忆存储（多租户隔离）            │
├─────────────────────────────────────────────┤
│  开发面（构建）                              │
│   · Agent 构建器（低代码/SDK）              │
│   · 工具市场、Agent 市场                     │
└─────────────────────────────────────────────┘
```

**AaaS 的架构挑战**：
- **沙箱多租户**：不同租户的 Agent 跑代码要强隔离（M9 沙箱 + 租户隔离叠加）
- **配额计费**：精确计量每租户的 token/计算/存储，转成账单
- **Agent 配置版本化**：用户改了 Agent 配置，要能回滚（L14-06 版本治理）
- **冷启动**：新租户的 Agent 首次调用要快，不能慢到流失

### 控制面 vs 数据面：平台架构核心模式

AaaS 的"控制面/数据面分离"是平台架构的核心模式，值得单独说清：

```
控制面（Control Plane）：管理操作
  · 创建/配置/删除 Agent
  · 租户管理、配额设置
  · 部署、监控
  · 低频、强一致

数据面（Data Plane）：运行时
  · 实际跑 Agent 请求
  · 检索、推理、工具执行
  · 高频、低延迟

为什么要分离：
  · 控制面操作（如部署新配置）不影响数据面正在服务的请求
  · 数据面扩容不波及控制面
  · 控制面挂了数据面还能继续服务（用旧配置）
```

> 这是云原生平台的标准架构（K8s 就是控制面/数据面）。Agent 平台沿用——管理操作和运行时服务分离，各自独立扩展、互不拖累。

### 平台架构的权衡

清醒看平台架构的代价——不是所有场景都该上平台：

```
该做平台的信号：
  · 多客户/多租户是核心需求
  · 要让用户自建 Agent（AaaS）
  · 规模大到单 Agent 架构不够
  · 商业模式是"卖 Agent 服务"

不该上平台的信号：
  · 就一两个内部 Agent 用 → 单 Agent 架构够
  · 没有多租户需求 → 别引入租户复杂度
  · 团队小扛不住平台运维
```

> 反模式：**过早平台化**。还没几个用户就上"多租户 AaaS 全套"——隔离/分片/计费/控制面数据面，复杂度爆炸，团队被基础设施拖垮，Agent 本身没做好。**先把单 Agent 做好、量来了再平台化**（L14-01 YAGNI 的延续）。

### 要点总结

- 从单 Agent 到平台是质变：操心点从"Agent 聪明"变"成百上千 Agent 怎么管/隔/扩/计费"
- 多租户是安全底线：数据/资源/权限三层隔离；串数据=灾难
- 数据隔离强制：所有查询带 tenant_id，封装租户感知接口禁止裸查（架构保证不可能忘过滤）
- 隔离三策略：逻辑(tenant_id最省)/Schema/物理(最强最贵)；常分级——小客户逻辑、大客户物理
- 水平扩展前提是无状态化：状态存外部(DB/Redis/向量库)，每请求加载写回；代价是多 IO 换可扩展+容错
- 扩展配套：负载均衡、分片(按tenant_id)、缓存、异步队列(长任务)
- AaaS=让用户构建部署自己的 Agent：控制面/数据面/开发面三层；挑战是沙箱多租户+配额计费+版本化+冷启动
- 控制面/数据面分离：管理操作和运行时服务分离，各自扩展互不拖累（云原生标准模式）
- 反模式过早平台化：没几个用户就上多租户AaaS全套→复杂度爆炸团队被拖垮；先做好单Agent量来了再平台化
