## Agent 评估方法论

到 M13 前，你的 Agent 已经能推理、用工具、有记忆、能多模态。但一个尴尬的问题：**它到底好不好？** 改了个 prompt，是变好了还是变差了？换了个模型，值不值？没有评估，所有"优化"都是拍脑袋。这一节先建立"怎么给会做决策的 Agent 打分"的方法论。

### 为什么 Agent 评估特别难

普通软件测试是"输入→断言输出"——给定输入，期望确定输出。Agent 把这套打碎了：

```
普通程序：确定性
  f(x) = y，每次都一样，assert f(2)==4

Agent：非确定性 + 多步决策
  · 同一问题，不同次运行可能走不同路径（选不同工具）
  · 输出是自然语言，没有"标准答案"可比对
  · "对"的标准模糊：完全对？部分对？答非所问？合理但非最优？
  · 一次失败可能是模型问题、工具问题、prompt 问题——难定位
```

**核心难题**：Agent 的输出**不是对错二值**，是**质量光谱**；且**过程和结果都重要**。给 Agent 打分，得同时看"最终答得好不好"和"中间走得对不对"。

### 结果评估 vs 轨迹评估

这是 Agent 评估最根本的二分：

```
结果评估（Outcome Eval）：
  只看最终输出对不对
  · 简单、自动、成本低
  · 缺点：忽略过程——蒙对了≠走对了；过程烂但结果对，下次未必
  · 适合：答案明确的任务（数学题、信息抽取）

轨迹评估（Trajectory Eval）：
  看中间每一步决策对不对（选对工具？调用顺序合理？有没有绕路？）
  · 全面、能定位"哪里烂"
  · 缺点：贵（要人工或LLM逐步判）、要有"参考轨迹"
  · 适合：复杂多步任务（研究、编码、运维）
```

| 维度 | 结果评估 | 轨迹评估 |
|------|---------|---------|
| 评估对象 | 最终输出 | 每步决策 |
| 成本 | 低 | 高 |
| 能否定位问题 | 否（只知烂，不知哪烂） | 是 |
| 需要参考 | 标准答案 | 参考轨迹 |
| 适合 | 答案明确任务 | 复杂多步任务 |

**生产建议**：**两者结合**。结果评估做大规模自动评测（便宜覆盖广），轨迹评估做关键案例深挖（贵但精准定位）。别只看结果——Agent 蒙对一次不代表可靠，轨迹评估能暴露"走对了流程但下次未必"的风险。

### 评估指标设计

具体打分打什么？按任务类型选指标：

```
1. 准确性（最核心）
   · 答案对不对、事实准不准
   · 指标：exact match / 事实正确率

2. 完整性
   · 该答的都答了吗？有没有遗漏
   · 指标：召回率（关键信息点覆盖率）

3. 工具使用质量（Agent 特有）
   · 选对工具？调用次数合理？有没有绕路？
   · 指标：工具选择准确率、平均步数、无效调用率

4. 安全性
   · 有没有越界/泄露/拒绝该答的
   · 指标：护栏触发率、安全违规率

5. 成本/延迟
   · Token 消耗、响应时间
   · 指标：avg tokens、p99 latency
```

> 关键认知：**Agent 评估不只看"对不对"**。一个答对但花了 20 步、烧了 1 万 token 的 Agent，比答对但 3 步、500 token 的差得多——生产要看"质量/成本/延迟"三角（呼应 M15）。

### 设计 Agent 评测集

评测集是评估的地基。一个有用的 Agent 评测集长这样：

```python
eval_cases = [
    {
        "id": "E001",
        "input": "帮我查北京明天天气",
        "expected_outcome": "返回北京明天的气温和天气状况",
        "expected_tools": ["weather_query"],   # 轨迹参考：该调哪个工具
        "eval_criteria": {
            "must_contain": ["北京", "明天"],   # 结果必须包含
            "must_not_contain": ["不知道", "无法"],  # 不能含
        },
    },
    {
        "id": "E002",
        "input": "总结这篇文档",   # 输入带附件
        "expected_outcome": "结构化摘要，覆盖主要论点",
        "expected_tools": ["file_read", "summarize"],
        "eval_criteria": {
            "min_length": 100,        # 摘要最低字数
            "fact_check": True,        # 需要事实校验（防编造）
        },
    },
]
```

**评测集设计原则**：
1. **覆盖典型场景**：常见用例必须有，别只测"能跑通的"
2. **包含边界/对抗案例**：模糊指令、越界请求、注入攻击——这些才暴露问题
3. **有明确评估标准**：must_contain / must_not_contain / 参考答案，让"好"可判定
4. **可回归**：案例固定，每次改 prompt/模型用同一套测，才能比变化
5. **分层**：简单功能案例（快测）+ 复杂综合案例（深测）

> 反模式：**用训练数据当评测集**——模型见过，测出来分数虚高。评测集必须是"模型没专门优化过"的真实分布。另一个反模式：**评测集太小或太单一**，10 个都是"打招呼"案例，覆盖率极低，改 prompt 全绿但实际质量崩。

### LLM-as-a-Judge：用模型评估模型

很多 Agent 输出没有标准答案，人工评又太贵。**LLM-as-Judge**——用另一个 LLM 当裁判给被测 Agent 的输出打分：

```python
from openai import OpenAI
judge_client = OpenAI()

def llm_judge(question: str, agent_answer: str, criteria: str) -> dict:
    """用 LLM 当裁判打分"""
    prompt = f"""你是严格的评估裁判。给 Agent 的回答打分。

问题：{question}
Agent 回答：{agent_answer}
评估标准：{criteria}

输出 JSON：
{{"score": 1-5, "reason": "打分理由", "issues": ["问题1", "问题2"]}}
评分要严格：5=优秀，3=及格有瑕疵，1=答非所问/错误。"""
    resp = judge_client.chat.completions.create(
        model="gpt-4o",   # 裁判用强模型
        messages=[{"role": "user", "content": prompt}],
        temperature=0,
        response_format={"type": "json_object"},
    )
    import json
    return json.loads(resp.choices[0].message.content)

# 用法
r = llm_judge("总结这篇文档", agent_output, "摘要应覆盖主要论点、无编造、100字内")
# {"score": 4, "reason": "覆盖了主要论点但漏了结论", "issues": ["遗漏结论"]}
```

**LLM-as-Judge 的要点**：
1. **裁判用更强模型**：被测用 mini，裁判用 4o——裁判要更准才可信
2. **裁判独立调用**：不和被测 Agent 共享上下文，避免"自己判自己"的偏袒
3. **评分标准明确**：给 criteria，"5=优秀 3=及格 1=差"，否则裁判尺度飘忽
4. **temperature=0**：裁判要稳定，同样的回答每次打分一致

### LLM-as-Judge 的偏倚与校准

LLM 当裁判不是没毛病——它有自己的偏倚：

```
常见偏倚：
  · 位置偏倚：把两个答案放一起评，模型倾向选第一个/第二个
  · 冗长偏倚：倾向给更长、更详细的答案高分（哪怕答非所问）
  · 自我偏倚：倾向认同"和自己风格像"的答案
  · 过于宽容：不愿打低分，常说"虽然...但也还行"

校准手段：
  · 位置交换：比较 A/B 时，交换顺序再评一次，看结论是否一致
  · 加约束：prompt 明确"长度不作为评分依据"
  · 人工锚点：抽一部分案例人工评，建立"基准分"，看 LLM 裁判和人工差多少
  · 多裁判投票：用多个模型/多次评，取多数（降单次偏倚）
```

> LLM-as-Judge 是"够用且省"的评估，不是"完美裁判"。生产里用它做大规模自动评，**配少量人工抽检校准**——别完全信任 LLM 裁判的每一个分，它也会偏。

### 评估的陷阱

```
陷阱1：过拟合评测集
  · 反复针对评测集调 prompt → 评测分涨但泛化崩
  · 对策：留出集（holdout）+ 真实线上数据抽查

陷阱2：只看平均分
  · 平均分 4.2，但某类案例全是 1 分被平均掩盖
  · 对策：按案例类别分桶看分布，别只看均值

陷阱3：只看成功不看成本
  · 答对了但烧了 10 万 token → 生产不可接受
  · 对策：质量+成本+延迟三角指标都看

陷阱4：一次性评估
  · 上线前评一次就不管 → 模型版本变了质量暗降
  · 对策：持续评测（L13-02 的 CI 评测）
```

### 要点总结

- Agent 评估难在：输出非确定、非二值对错、过程和结果都重要
- 根本二分：结果评估（只看输出，便宜覆盖广）vs 轨迹评估（看每步决策，贵但能定位）——生产两者结合
- 指标不只看"对不对"：准确性+完整性+工具质量+安全性+成本延迟（质量/成本/延迟三角）
- 评测集设计：覆盖典型+边界对抗、明确评估标准、可回归、分层；禁用训练数据当评测
- LLM-as-Judge：用更强模型当裁判，独立调用、明确评分标准、temperature=0
- LLM 裁判偏倚：位置/冗长/自我偏倚/过宽容；校准靠交换顺序、加约束、人工锚点、多裁判投票
- 陷阱：过拟合评测集、只看平均分、只看成功不看成本、一次性评估——分别对策
